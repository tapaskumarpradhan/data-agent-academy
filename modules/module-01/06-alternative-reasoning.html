Alternative Reasoning Patterns

ReAct is the default, but it's not the only way agents can think. Four distinct reasoning patterns serve different problem types — and production systems often combine them.

REASONING PATTERNS
Core Patterns

Four Ways Agents Reason

01
ReAct (Reason + Act)
Think → Act → Observe → Repeat

How it works: The agent reasons about what to do, takes an action (tool call), observes the result, then decides the next step. This cycle repeats until the agent has enough information to respond.

Best for: General-purpose tool-use tasks. Data lookups, API calls, question-answering with retrieval. The bread and butter of most production agents.

Weakness: Can get stuck in loops — retrying the same failed approach repeatedly. No explicit planning phase means the agent may take inefficient paths through complex multi-step problems.

General Purpose Low Cost Loop Risk
02
Plan-and-Execute
Plan (Full) → Execute Step 1 → Execute Step 2 → ... → Replan?

How it works: A Planner LLM creates a complete step-by-step plan before any execution begins. An Executor LLM (often a smaller, cheaper model) carries out each step. After execution, the planner can optionally revise the remaining plan based on results.

Best for: Complex multi-step workflows where the path needs to be coordinated upfront — generating reports, running multi-stage analyses, building dashboards. The explicit plan provides transparency and predictability.

Weakness: Rigid. The initial plan may not account for surprises. Replanning adds latency. If the planner misunderstands the task, every execution step carries the error forward.

Complex Workflows Predictable Rigid Plans
03
Tree-of-Thought (ToT)
Generate 3 Paths → Evaluate Each → Pursue Best → Branch Again?

How it works: Instead of following a single reasoning path, the agent generates multiple candidate approaches simultaneously. Each path is evaluated (often by the same LLM acting as a critic), and the most promising path is pursued further. Unsuccessful paths are pruned. This creates a tree structure of explored reasoning.

Best for: Complex analytical problems where multiple valid approaches exist. Root cause analysis, optimization problems, strategic planning. The breadth of exploration often finds better solutions than single-path reasoning.

Weakness: Expensive. Generating and evaluating multiple paths multiplies token usage by 3-5x compared to ReAct. Latency increases significantly. Only justified for high-value decisions where solution quality matters more than speed or cost.

High-Value Decisions Best Quality High Cost
04
Reflexion
Execute → Reflect → Learn → Retry

How it works: The agent executes a task, then explicitly reflects on the result: What worked? What failed? What should be different next time? This self-critique produces a concrete learning insight that informs the next attempt. Unlike simple retry, Reflexion creates genuine self-improvement loops.

Best for: Tasks where learning from failure materially improves results. Debugging code, investigating anomalies, iterating on data analysis. The reflection step transforms failures from dead ends into stepping stones.

Weakness: Higher latency due to the reflection overhead on every iteration. The reflection quality depends on the LLM's self-assessment capability. Can over-correct if the reflection misdiagnoses the failure.

Investigation Self-Improving Higher Latency
Comparison

When to Use Each Pattern

Pattern Best For Avoid When Cost Latency
ReAct General tool-use, data lookups, simple Q&A Complex multi-step tasks with dependencies Low Low
Plan-and-Execute Reports, multi-stage analyses, coordinated workflows Dynamic tasks where the path can't be predicted Medium Medium
Tree-of-Thought Root cause analysis, optimization, strategic decisions Simple lookups, cost-sensitive operations High (3-5x) High
Reflexion Debugging, anomaly investigation, iterative analysis Tasks with strict latency requirements Medium-High High
Production Reality
Start with ReAct as your default. Add Plan-and-Execute for complex multi-step workflows where upfront coordination pays off. Use Tree-of-Thought and Reflexion selectively for high-value tasks where quality justifies the cost and latency overhead. Most production agents use ReAct for 80% of queries, with specialized patterns triggered by task complexity detection.
Advanced Strategy

Combining Patterns in Production

Production agents don't pick one pattern and use it forever. The most effective architectures route queries to different reasoning patterns based on complexity analysis. A lightweight classifier (often a small LLM call or heuristic) examines the incoming query and selects the appropriate pattern.

Complexity Router

A classifier assigns each query a complexity tier: Simple (ReAct), Complex (Plan-and-Execute), Investigative (Reflexion), or Strategic (Tree-of-Thought). This routing happens in under 500ms and saves significant cost by avoiding expensive patterns for simple queries.

Pattern Escalation

Start with ReAct. If the agent loops 3 times without progress, automatically escalate to Plan-and-Execute. If the plan fails, escalate to Reflexion. This graceful degradation ensures simple queries stay fast while complex ones get the reasoning depth they need.

Case Study

AcmeCorp: Pattern Selection in Action

AcmeCorp's data agent receives queries of vastly different complexity. Here's how each reasoning pattern handles a real scenario from their retail analytics platform.

ReAct Pattern
"Show me revenue for Q4"

Simple data retrieval. The agent thinks (need Q4 revenue data), acts (runs SQL query), observes (gets results), and responds. One cycle, under 3 seconds. ReAct handles this perfectly — no planning overhead needed.

Plan-and-Execute Pattern
"Create a quarterly business review with charts comparing all regions against targets"

Complex multi-step workflow. The planner creates a 6-step plan: (1) fetch regional targets, (2) pull actual revenue by region, (3) calculate variances, (4) generate comparison charts, (5) identify top and bottom performers, (6) compile into formatted report. The executor processes each step sequentially, with the planner monitoring progress and adjusting if a data source is unavailable.

Reflexion Pattern
"Investigate why Southwest revenue dropped 30%"

Investigative task. First attempt: checks if stores closed — no closures found. Reflects: store count is stable, so the issue is per-store performance. Second attempt: examines product category trends — finds electronics dropped 45%. Reflects: electronics is the driver, but why? Third attempt: correlates with a competitor store opening. The reflection step transformed each dead end into a more targeted hypothesis, converging on the root cause in 3 iterations.

Pattern Selection Heuristic
AcmeCorp's complexity router uses a simple heuristic: if the query contains action verbs like "show" or "get" with a single metric → ReAct. If it mentions "compare," "review," or "report" with multiple dimensions → Plan-and-Execute. If it asks "why" or "investigate" → Reflexion. This simple classification correctly routes 85% of queries to the optimal pattern.
Implementation Notes

Production Implementation Tips

Setting Max Iterations per Pattern

ReAct: Cap at 5-8 iterations. If the agent hasn't converged by then, it's likely stuck in a loop. Escalate to Plan-and-Execute.

Plan-and-Execute: Maximum 10 plan steps. Longer plans degrade in quality. If a task genuinely needs more than 10 steps, decompose it into sub-tasks.

Tree-of-Thought: Generate 3 branches maximum. More branches rarely improve quality but always increase cost. Prune aggressively after evaluation.

Reflexion: Maximum 3 reflection cycles. Diminishing returns after the third attempt. If still failing, surface the partial result to the user with a transparency note.

Measuring Pattern Effectiveness

Track three metrics per pattern: success rate (did the agent produce a correct result?), cost per query (total tokens consumed), and latency (wall-clock time). Plot these on a weekly dashboard. If a pattern's success rate drops below 70%, investigate whether queries are being misrouted or if the underlying data has changed.
