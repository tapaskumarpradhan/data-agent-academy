Module 01 — Lesson 01

The Five-Layer
Architecture

Why a chatbot wrapper is not enough — and the layered blueprint that makes a Data Agent production-ready.

The Problem

Why Not Just Wrap a Chatbot?

AcmeCorp's CEO walks into the all-hands: "We're paying for an LLM already. Why can't we just plug ChatGPT into our database and let people ask questions?"

It sounds reasonable. But here is what breaks in the first week:

1 Hallucinated SQL
The LLM invents column names that do not exist. sales.revenue looks plausible but the actual column is sales.total_amount.
2 No Access Control
A store manager asks about payroll and gets the full company payroll. There is no concept of who can see what.
3 Untracked Quality
Nobody knows if the answers are right. The agent gives confident wrong answers that go into board decks unchecked.
AcmeCorp Reality Check
Week one: a regional manager asks "show me revenue for the Southwest region" and gets data for all regions because there is no guardrail enforcing their scope. The CEO is not happy.

A real Data Agent needs five distinct layers working together. Strip any one of them out and the system fails in production. Let us walk through each.

L1
Layer 1
Agent Runtime

The runtime is the brain of the agent. It receives the user's natural language question, decides what steps to take, maintains conversation state, and orchestrates calls to every other layer.

Think of it as the project manager: it never touches the database directly but knows exactly who to call and in what order.

LLM Core State Manager Memory Orchestrator
AcmeCorp Scenario
A product manager types: "Show me Q3 revenue by region." The Agent Runtime receives this, recognizes it as a data query, plans the steps (discover schema, build SQL, execute, format), and orchestrates everything that follows.
L2
Layer 2
Tool Layer

Tools are the actions the agent can actually perform: execute a SQL query, call an API, read a file, or perform a calculation. Each tool has a strict interface — a name, description, parameters, and return type.

Without well-designed tools, the LLM is just a brain with no hands.

query_execute schema_discovery api_fetch calculate
AcmeCorp Scenario
The agent connects to AcmeCorp's PostgreSQL warehouse, Shopify API for live inventory, and shared Excel files for manual adjustments. Each connection is a discrete, well-defined tool.
L3
Layer 3
Data & Context (RAG)

This layer gives the agent semantic understanding of your data. Through retrieval-augmented generation (RAG), the agent knows your schema, your business glossary, and the meaning behind column names.

It is the difference between an agent that guesses column names and one that knows them.

Schema Embeddings Business Glossary Query Examples Metadata
AcmeCorp Scenario
When someone says "revenue," the RAG layer knows this means sales.total_amount, not sales.gross_amount. It also knows that "region" maps to stores.sales_region and that Q3 means July through September for AcmeCorp's fiscal calendar.
L4
Layer 4
Guardrails

Guardrails enforce who can see what, what queries are allowed, and what data is redacted. This includes role-based access control (RBAC), PII masking, query complexity limits, and approval workflows for sensitive operations.

This is the layer that turns a fun demo into an enterprise-grade tool.

RBAC PII Masking Rate Limits Approval Flows
AcmeCorp Scenario
Store managers can only see data for their own store. Regional managers see their region. Nobody sees raw PII like customer emails or credit card numbers — those are automatically masked before results reach the user.
L5
Layer 5
Evaluation

How do you know the agent is working correctly? The evaluation layer tracks accuracy, response time, LLM token cost, user satisfaction, and error rates. It is the feedback loop that drives continuous improvement.

Without evaluation, you are flying blind — confident that everything works until a board presentation goes wrong.

Accuracy Score Latency P95 Token Cost User Feedback
AcmeCorp Scenario
AcmeCorp tracks three key metrics: Is the data correct? (validated against known queries), How fast is the response? (target under 5 seconds), and What does it cost? (LLM spend per query averaged across departments).
Real-World Parallels

You Have Seen This Pattern Before

The five-layer architecture is not unique to Data Agents. Every serious platform separates concerns the same way.

Netflix: From Monolith to Layered Microservices ▶

Netflix started as a monolithic application. As they scaled, they split into layers: an API Gateway (like the runtime), microservices (like tools), a data platform (like the context layer), access policies (like guardrails), and a chaos engineering suite (like evaluation).

The lesson: when you strip a layer away, the whole system becomes fragile. Netflix learned this by intentionally breaking things. You can learn it here without the production incidents.

Uber: Orchestration Under Extreme Load ▶

Uber's architecture separates the dispatch brain (runtime) from maps, pricing, and ETA services (tools), a geospatial context layer, fraud and safety guardrails, and a real-time analytics evaluation layer.

AcmeCorp's Data Agent follows the same pattern at a smaller scale. The runtime orchestrates, tools execute, context informs, guardrails protect, and evaluation measures.
