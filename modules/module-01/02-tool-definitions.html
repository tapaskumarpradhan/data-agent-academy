Tool Definitions

How to give a Data Agent hands — the anatomy of well-defined, composable tools that turn LLM reasoning into real-world actions.

The Scene

Tracing a Real Question

An AcmeCorp analyst opens the Data Agent and types:

"What were our top 10 products last quarter?"

Before the agent can answer, it needs to figure out what to do. That means identifying the right sequence of tools to call. Let us trace the flow:

schema_discovery → query_execute → aggregate → calculate → export_csv
AcmeCorp Trace
First, the agent discovers which tables exist (products, sales, stores). Then it builds and executes a SQL query joining sales to products. It aggregates by product_name, calculates each product's share of total revenue, and finally exports the result as CSV for the analyst.
Tool Categories

Five Types of Tools

Every Data Agent tool falls into one of five categories. Each category has a distinct purpose.

Data Access

Read from and discover data sources. These are the foundational tools for any data agent.

schema_discovery query_execute table_metadata
AcmeCorp
Finds the products, sales, and stores tables and reads their column types.
Transformation

Reshape, combine, and restructure data after retrieval. The glue between raw rows and useful answers.

filter aggregate join pivot
AcmeCorp
Joins sales to products, aggregates by product_name, and sorts descending.
External APIs

Reach beyond the warehouse to third-party systems for live data, notifications, and integrations.

slack_notify jira_create api_fetch
AcmeCorp
Pulls current inventory counts from the Shopify API to cross-reference with sales data.
Computation

Perform math, validation, and comparisons that the LLM should not attempt itself.

calculate validate compare
AcmeCorp
Calculates each product's percentage of total revenue. LLMs are bad at precise arithmetic — tools are not.
File Operations

Read from and write to files — ingest uploaded data and export formatted results.

read_csv write_json export_pdf
AcmeCorp
Exports the final top-10 products result as a CSV file that the analyst can download.
Anatomy of a Tool

The JSON Contract

Every tool is defined by a strict JSON schema that the LLM reads to understand what the tool does, what inputs it needs, and what it returns. This is the contract between the brain and the hands.

tool_definition.json
{ "name": "query_execute", "description": "Execute a read-only SQL query against the data warehouse and return results as JSON rows.", "parameters": { "type": "object", "properties": { "sql": { "type": "string", "description": "The SQL SELECT statement to execute" }, "timeout_ms": { "type": "integer", "default": 5000, "description": "Max execution time in milliseconds" }, "max_rows": { "type": "integer", "default": 1000, "description": "Maximum rows to return" } }, "required": ["sql"] }, "returns": { "type": "object", "properties": { "rows": { "type": "array" }, "row_count": { "type": "integer" }, "execution_ms": { "type": "integer" } } } }

Notice the four key parts: name (unique identifier), description (what the LLM reads to decide when to use it), parameters (JSON Schema for inputs), and returns (what the caller gets back).

Design Principle

Narrow Tools Beat God Tools

The single biggest tool design mistake is building one monolithic do_everything() function. Narrow, composable tools give the LLM more flexibility and produce fewer errors.

Bad: God Tool
anti-pattern.json
{ "name": "do_everything", "description": "Answer any data question", "parameters": { "query": "string" } }

Hides all complexity inside one function. The LLM cannot reason about intermediate steps, retry partial failures, or compose with other tools.

Good: Composable Tools
composable-tools.json
// 5 narrow, focused tools "schema_discovery" // find tables "query_execute" // run SQL "aggregate" // group & sum "calculate" // math ops "export_csv" // output file

Each tool does one thing. The LLM can chain them, retry individually, and the system is debuggable and testable.

Design Rule
If a tool description needs the word "and" more than once, it is doing too much. Split it. An agent with 20 narrow tools outperforms one with 3 broad tools every time.
