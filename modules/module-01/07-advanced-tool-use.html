Module 02 — Lesson 07

Advanced Tool Use Patterns

The 2025 frontier of agent tool use goes far beyond basic function calling. These six patterns solve the scaling challenges that break naive implementations.

ADVANCED TOOL USE PATTERNS
Pattern 01

Tool Search (Dynamic Discovery)

When an agent has access to 100+ tools, loading all tool schemas into the context window becomes prohibitively expensive — consuming thousands of tokens before the agent even starts reasoning. Tool Search solves this with a meta-tool: a lightweight search interface that discovers relevant tools dynamically.

Without Tool Search

Load all 150 tool schemas upfront. Consumes ~12,000 tokens of context. LLM sees irrelevant tools, leading to confusion and wrong tool selection. Context window fills up before reasoning begins.

With Tool Search

Load only 1 meta-tool (search_tools). Agent searches for relevant tools on demand. Only 3-5 relevant schemas loaded per query. Context stays clean. Tool selection accuracy improves dramatically.

85% Context reduction
3-5 Tools loaded per query
+40% Selection accuracy
Implementation
The meta-tool uses semantic search over tool descriptions. When the agent calls search_tools("revenue by region"), it returns the top 3-5 matching tool schemas based on embedding similarity. The agent then selects and invokes the appropriate tool with full schema awareness.
Pattern 02

Programmatic Tool Calling

Standard tool calling is one-at-a-time: the agent calls a tool, waits for the result, reasons, calls the next tool. For batch operations, this creates enormous round-trip overhead. Programmatic tool calling lets agents invoke tools within code execution — with loops, conditionals, and error handling.

How It Works

Instead of the LLM deciding each tool call individually, it writes a code block that orchestrates multiple tool calls programmatically. The code execution environment provides the tool functions as callable APIs. The agent can use loops to iterate over datasets, conditionals to branch on results, and try/catch blocks for error handling.

# Agent writes this code block instead of individual tool calls
results = []
for store_id in range(1, 201):
    data = query_store(store_id, "revenue_q4")
    if data.revenue < threshold:
        results.append(data)
# 200 queries in 1 code execution, not 200 LLM round-trips
Sequential: 200 Round-Trips

Each store query requires an LLM reasoning step. 200 stores = 200 think-act-observe cycles. Total time: ~10 minutes. Token cost: ~400K tokens.

Programmatic: 1 Code Block

Agent writes one loop that calls the tool 200 times within code execution. Total time: ~15 seconds. Token cost: ~2K tokens for the code + tool results.

Pattern 03

Tool Use Examples (Few-Shot)

Tool descriptions tell the agent what a tool does. Tool examples show the agent how to use it. Providing 2-3 usage examples with each tool definition dramatically improves tool selection accuracy and parameter formatting — especially for tools with complex parameter schemas.

Description Only

Tool description says: "Query sales data with filters." Agent guesses parameter format. Wrong 30% of the time for complex filters. Parameters are often malformed or missing required fields.

+ 1 Example

One example shows the basic parameter format. Error rate drops to 15%. Agent correctly formats simple queries but still struggles with edge cases like date ranges and nested filters.

+ 3 Examples

Three examples cover simple, moderate, and complex use cases. Error rate drops to 5%. Agent generalizes the pattern to novel queries. The examples act as implicit documentation that the LLM actually understands.

Best Practice
Choose examples that cover different complexity levels: one simple case, one with optional parameters, and one with edge cases (null handling, date ranges, aggregations). Keep examples concise — the goal is to show the pattern, not exhaustive documentation. 2-3 examples add roughly 200-400 tokens per tool but save thousands of tokens in retries.
Pattern 04

Agent Skills (Open Standard)

Agent Skills are reusable capability packages that bundle tools, prompts, and configuration into a single deployable unit. Think of them as plugins for AI agents. A skill like "data-quality-audit" packages the SQL tools, validation logic, quality metrics, and reporting templates into one installable component.

The Skill Ecosystem

First-party skills: Built by framework providers (Anthropic, OpenAI, Google) as reference implementations. Cover common capabilities like web search, code execution, file management, and data analysis.

Partner-built skills: Third-party skill directories are emerging, similar to package registries (npm, PyPI). Organizations publish and share domain-specific skills. A "retail-analytics" skill from a partner could give any agent instant access to retail KPI calculations, seasonal adjustment algorithms, and standard report templates.

Custom skills: Organizations build internal skills that encode proprietary knowledge. AcmeCorp's "inventory-optimization" skill bundles their specific reorder algorithms, supplier preferences, and safety stock calculations.

Pattern 05

On-Demand MCP Tool Loading

MCP (Model Context Protocol) provides a standard interface for tool integration. On-demand loading takes this further: instead of connecting to all MCP servers at startup, the agent defers tool definitions until they're actually needed. This lazy-loading approach reduces initial context consumption to near zero.

How Deferred Loading Works

Step 1 — Registry: At startup, the agent loads only a lightweight tool registry — a catalog of available MCP servers with one-line descriptions. This consumes ~500 tokens regardless of how many tools exist.

Step 2 — Discovery: When the agent needs a capability (e.g., "query the inventory database"), it searches the registry and identifies the relevant MCP server.

Step 3 — Connection: The agent connects to the MCP server on demand, loading only that server's tool schemas into context. Connection is maintained for the session duration to avoid reconnection overhead.

Step 4 — Release: After the task is complete, unused MCP connections can be released to free context space for subsequent tasks.

~500 Initial token cost
On-demand Schema loading
50+ Scalable tool count
Pattern 06

Token-Efficient Tool Design

Every token in a tool schema or tool result costs money and consumes context. Token-efficient design minimizes waste without sacrificing the agent's ability to use tools correctly. Three strategies make the biggest impact.

Minimize Descriptions

Write tool descriptions like API docs, not essays. Remove filler words, redundant explanations, and marketing language. A 200-token description can usually be trimmed to 50 tokens without losing information the LLM needs for tool selection.

Return Essential Fields

Tool results should return only the fields the agent needs. If a database query returns 20 columns but the agent only needs 3, filter at the tool layer. Returning a full ORM object when you need a count wastes hundreds of tokens per call.

Use Pagination

Never return 10,000 rows in a single tool result. Implement pagination with a default page size of 10-25 results. The agent can request more if needed. This prevents context window overflow and keeps reasoning focused on manageable data volumes.

Token Budget Calculation

For a 128K context window agent, budget your tokens carefully:

System prompt & instructions: ~2,000 tokens (fixed overhead)

Tool schemas: ~50-100 tokens per tool. With 5 active tools: ~500 tokens.

Conversation history: Reserve 30-50% for multi-turn conversations: ~40-64K tokens.

Tool results: Budget 1-2K tokens per tool call result. Plan for 5-10 calls per interaction: ~10-20K tokens.

Reasoning space: The remaining tokens are for the LLM's chain-of-thought. Never consume so much context that the agent has no room to think.

Case Study

AcmeCorp: Scaling to 50+ Data Sources

AcmeCorp's data agent connects to 50+ data sources: inventory databases, sales APIs, CRM systems, supplier portals, and regional data warehouses. The naive approach of loading all 50+ tool schemas at startup consumed 40% of their context window before the agent processed a single query.

AcmeCorp's Advanced Tool Architecture

Tool Search for Discovery: A meta-tool indexes all 50+ data sources by semantic description. When the agent receives a query about "supplier lead times," Tool Search returns schemas for the Supplier Portal API and the Logistics DB — not the 48 other irrelevant tools.

MCP for On-Demand Loading: Each data source is wrapped in an MCP server. The agent connects only to the 2-3 servers needed for the current query. Startup context cost dropped from ~15,000 tokens to ~500 tokens.

Programmatic Batch Operations: Checking data quality across all 200 stores used to require 200 sequential LLM reasoning cycles (one per store). With programmatic tool calling, the agent writes a single loop that queries all 200 stores in one code execution block. Processing time dropped from 10 minutes to 15 seconds. Token cost dropped from 400K to 2K.

Few-Shot Examples: Each tool includes 2-3 usage examples. SQL query formatting errors dropped from 25% to under 3%. The examples were worth 600 tokens of context but saved thousands of tokens in retry loops.

Result
By combining these patterns, AcmeCorp reduced their average query cost by 70%, cut response time by 60%, and eliminated the context overflow errors that plagued their initial deployment. The key insight: advanced tool use isn't about more tools — it's about smarter tool management.
