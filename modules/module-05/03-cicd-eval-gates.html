CI/CD PIPELINE WITH EVAL GATES
The Scene

The Prompt Update That Took Down 200 Stores

AcmeCorp’s data team updated the Data Engineer agent’s system prompt to improve SQL accuracy. The change looked harmless — three words added to a tool description. No one ran the golden dataset. No evaluation gate existed. The update went straight to production.

Incident Report
Wednesday, 11:15 AM — After the prompt update, the Data Engineer agent began selecting query_execute instead of schema_discovery for metadata requests. This triggered raw SQL against production tables without schema validation. Within 2 hours, three store managers received incorrect inventory counts. The regional VP made a restocking decision based on hallucinated data. The agent had to be rolled back — but not before $47,000 in unnecessary inventory orders were placed across 14 stores.

The root cause? No evaluation gate. A three-word prompt change altered tool selection behavior, and nobody caught it before production. This module teaches you how to prevent exactly this.

The Evolution

DevOps → MLOps → AgentOps

Each evolution adds new complexity. Agents are the hardest to deploy safely because they modify persistent data across multiple systems.

DisciplineWhat You DeployKey Challenge
DevOps Deterministic code Same input → same output. Rollback is straightforward.
MLOps Probabilistic models + code Model drift, data drift, A/B testing required.
AgentOps Autonomous systems + models + tools + prompts + guardrails Non-deterministic behavior, persistent side effects, multi-component versioning.
AcmeCorp Reality
Remember the ALTER TABLE incident from Module 03? The Data Engineer agent ran ALTER TABLE customers on production, causing a 4-hour outage across 200 stores. That was a guardrail failure. The prompt update incident was an AgentOps failure — the agent’s behavior changed without anyone testing whether the change was safe. Both are preventable with the right infrastructure.
The Pipeline

CI/CD Pipeline for Agents

The pipeline has familiar stages — plus one critical addition: the Evaluation Gate, which blocks deployment if quality metrics drop below thresholds.

CI Continuous Integration
  • Unit tests for tool functions
  • Prompt regression tests
  • Tool selection accuracy tests
  • Guardrail validation
  • Cost estimation (will this change increase spend?)
CD Continuous Delivery
  • Automated packaging of all components
  • Environment-specific configuration
  • Canary or shadow deployment to staging
  • Monitor metrics for 24-48 hours
  • Gradual rollout: 5% → 25% → 50% → 100%
The Key Differentiator

Evaluation Gate: The Agent Quality Firewall

This is the stage that separates AgentOps from traditional CI/CD. The evaluation gate runs your golden dataset (50-200 test cases) against the new agent version and blocks deployment if quality drops below thresholds.

Evaluation Gate Configuration

evaluation_gate:
  blocking:                          # Any failure blocks deployment
    tool_selection_accuracy: 0.95     # Must select correct tool 95%+ of time
    correctness_score: 0.90          # LLM-as-judge accuracy threshold
    safety_violations: 0             # Zero tolerance for PII leaks or policy violations
    p99_latency_seconds: 45          # Worst-case must be under 45s
    cost_per_task_multiplier: 2.0    # Cannot exceed 2x current production cost

  warning:                            # Flags for review, does not block
    hallucination_rate: 0.01         # Alert if hallucination exceeds 1%
    tool_efficiency_ratio: 1.8       # Agent using too many tool calls
    context_utilization: 0.85        # Approaching context window limits

  on_failure:
    action: halt_pipeline
    notify:
      - slack: "#agent-deployments"
      - email: "agentops-team@company.com"
    report: detailed               # Shows which metrics failed and by how much
