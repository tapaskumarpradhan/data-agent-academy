Ship agent updates safely and track every moving piece — because a prompt change can break things just as badly as a code change.

DEPLOYMENT STRATEGIES
The Scene

The Version Mismatch That Hallucinated Table Names

After the $47K prompt update incident (Lesson 03), AcmeCorp’s data team updated the Data Engineer agent’s prompt to fix schema_discovery vs query_execute selection. The fix worked great — for two weeks. Then a different engineer reverted the LLM model from Claude 3.5 to Claude 3 to cut costs. But the new prompt was written for Claude 3.5’s instruction-following capabilities and was incompatible with Claude 3.

The agent started hallucinating table names, referencing sales.revenue instead of the correct sales.total_amount. Nobody connected the dots because the prompt change and the model change were tracked in different systems.

Incident Report
Thursday, 10:15 AM — The Data Engineer agent begins returning queries against sales.revenue, a table that does not exist. The LLM confidently fabricates column names. Three analysts build reports on the hallucinated data before anyone notices. Root cause: prompt v14 was designed for Claude 3.5’s structured output format, but the model had been silently downgraded to Claude 3. AcmeCorp needed a unified version manifest.
Deployment Patterns

Four Ways to Ship Agent Updates

1 Canary Deployment Gradual Rollout

Route 5% of traffic to the new version. Monitor for 24–48 hours. If metrics hold, gradually increase to 25%, 50%, then 100%. If anything degrades, roll back instantly with zero user impact.

AcmeCorp Implementation
After implementing canary deploys, AcmeCorp routes new agent versions to the Southwest region first (their smallest, 22 stores). The QA agent’s 47 daily validation checks run against both old and new versions. If accuracy drops or the $0.004/query cost target is exceeded, traffic stays at 5% and the team investigates.
2 Shadow Mode Zero Risk Testing

New agent runs alongside old one. Both process every request, but only the old version’s output is shown to users. The new version’s outputs are logged and compared offline — perfect for testing major changes without any user-facing risk.

AcmeCorp Implementation
When AcmeCorp added chain-of-thought reasoning to the PO agent, they ran it in shadow mode for one week. Both the old and new PO agents answered the VP of Sales’ queries. The team compared outputs offline. The new version was more complete but 3x slower — they optimized before going live.
3 Blue-Green Deployment Instant Rollback

Two identical environments. Switch atomically. The “blue” environment runs the current version while “green” runs the new one. A load balancer flips traffic from blue to green. If anything goes wrong, flip back in seconds.

AcmeCorp Implementation
AcmeCorp uses blue-green for the Data Steward agent because it handles compliance checks. If the new version has a bug in PII detection, they can switch back to the old version in under 60 seconds. No compliance gap.
4 Feature Flags Per-User Control

Enable or disable agent behaviors without redeployment. Wrap new capabilities in conditional flags. Toggle them on for specific users, teams, or regions — no code deploy needed.

AcmeCorp Implementation
When adding Shopify table support, AcmeCorp used a feature flag. The Data Engineer agent could toggle between “Shopify-enabled” and “warehouse-only” mode. This let them test Shopify integration with 3 power users before enabling it for everyone.
Rollback Complexity

Why Agent Rollback Is Harder Than Traditional Software

Traditional software rollback is straightforward: deploy the previous version. Agent rollback is fundamentally different because agents take actions in the real world. They write to databases, send messages, and call external APIs. You can roll back the agent, but you cannot un-send an email or un-create a Jira ticket.

AcmeCorp Lesson Learned
When AcmeCorp rolled back the prompt update, the damage was already done — 14 stores had placed inventory orders based on hallucinated data. Rolling back the agent didn’t roll back the purchase orders.
Component Rollback Difficulty Why
Agent code Easy Just deploy previous version
Prompts Easy Version-controlled, swap back
Model version Medium May need re-testing with current prompts
Database writes Hard Agent already INSERT’d/UPDATE’d records
Sent messages Impossible Slack notifications, emails already delivered
External API calls Impossible Jira tickets created, orders placed
Key Takeaway
The best rollback strategy is a good canary strategy — catch problems before they affect most users.
Version Everything

Seven Things to Version

An agent is not just code. It’s code + prompts + model + config + data + tools + guardrails. Change any one piece and you have a different agent. Version all seven.

💻
Code
Git + SemVer
✍
Prompts
LangSmith / PromptLayer
A prompt change altered AcmeCorp’s tool selection behavior as much as a code change would.
🤖
Models
MLflow / Registry
The Claude 3.5 → Claude 3 revert caused hallucinated table names.
⚙
Configuration
Config-as-code
Temperature, max_tokens, top-k per persona.
🗂
Data / Embeddings
DVC / LakeFS
When Shopify tables were added, the RAG embeddings needed updating.
🔧
Tool Schemas
Schema Registry
MCP server definitions change when tools are added or modified.
🛡
Guardrail Policies
OPA / Git
The PII masking rules that prevent customer email exposure.
AcmeCorp Solution
AcmeCorp now pins all 7 components in a single deploy manifest. Every release includes: code hash, prompt hash, model ID, tool schema version, guardrail policy version, embedding version, and config hash. The manifest is the single source of truth. You can reproduce any past agent state by deploying its manifest.
acmecorp-agent-v2.4.1:
  code: git@abc123
  prompt: prompt-v14 (sha: def456)
  model: claude-3.5-sonnet-20250601
  tools: mcp-schema-v3.2
  guardrails: policy-v2.1
  embeddings: embed-20250315
  config: temp=0.1, max_tokens=2048

The Bottom Line
40%+ of agentic AI deployments will be canceled by 2027 due to costs, unclear value, or poor risk controls (Gartner). AcmeCorp was on track to be in that 40% after three production incidents in their first month. The evaluation gate, version pinning, and canary deployment infrastructure cost two weeks of engineering time — but prevented an estimated $200K+ in potential damages over the following quarter.
