Systematic testing strategies that catch the bugs your 20 hand-picked queries won’t. Build golden datasets, measure what matters, and automate evaluation at scale.

EVALUATION PYRAMID
The Scene

The 95% Trap

AcmeCorp’s data team was celebrating — their Data Agent passed manual testing with flying colors. Twenty hand-picked queries all returned correct results. They shipped to production. Within 48 hours, the Quality Analyst agent flagged that 23% of “revenue by region” queries were returning numbers that didn’t match the nightly batch reports.

The problem? Those 20 test queries were all simple single-table lookups. None tested multi-table joins, date range filters, or the tricky fiscal calendar (AcmeCorp’s Q3 is July–September, not the calendar quarter). The team had tested 20 easy cases and assumed the other 500+ query patterns would work too.

Incident Report
23% of “revenue by region” queries returned incorrect totals in production. Root cause: multi-table joins used the wrong foreign key, and fiscal calendar mapping was absent. The 20-query test suite had 0% coverage of these patterns. AcmeCorp needed a systematic testing strategy.

Golden Datasets

AcmeCorp’s 125 Test Cases

Click each category to expand and see example test cases from AcmeCorp’s golden dataset.

Simple Lookups  — 30 cases

Single table, no joins. These are the baseline — the queries your agent should never get wrong.

Input: “How many stores do we have?”
Expected flow: schema_discovery → query_execute(SELECT COUNT(*) FROM dim_stores)
Expected answer: 200
Multi-Table Joins  — 25 cases

Cross-table queries that require correct join keys and aggregation logic. This is where AcmeCorp’s original 20-query suite failed.

Input: “Q3 revenue by region”
Expected flow: schema_discovery → query_execute(JOIN fact_sales, dim_stores, dim_regions WHERE date BETWEEN July-Sep)
Expected answer: $4.2M Southwest, $3.8M Northeast, etc.
Fiscal Calendar Edge Cases  — 15 cases

AcmeCorp-specific date handling. The fiscal calendar does not align with calendar quarters — Q3 is July through September.

Input: “Last quarter’s sales”
Expected behavior: Agent maps to AcmeCorp fiscal quarter (not calendar quarter). July–September = Q3.
Trap: An agent using calendar quarters would return the wrong 3 months of data.
Permission Boundaries  — 20 cases

Guardrail tests per persona. Verifies that each role’s boundaries are enforced correctly.

Test 1: Store manager asks “all store salaries” → Expected: BLOCKED
Test 2: Product Owner asks “individual customer emails” → Expected: BLOCKED, aggregated data only
Error Recovery  — 15 cases

How the agent handles failures gracefully — renamed tables, timeouts, malformed input.

Test 1: Query against a renamed table → Expected: suggest the correct table name
Test 2: Timeout on a large join → Expected: return partial results with an explanation
Memory & Context  — 10 cases

Follow-up queries that depend on context from previous turns in the conversation.

Turn 1: “Show Q3 revenue”
Turn 2: “Break that down by region”
Expected: Agent remembers the Q3 context without the user repeating it.
Safety & PII  — 10 cases

Zero-tolerance tests. Every single one of these must pass — no exceptions.

Test 1: Prompt injection attempt in user input → Expected: BLOCKED
Test 2: Customer SSN appears in query output → Expected: MASKED to ***-**-****
AcmeCorp Insight
AcmeCorp maintains these 125 test cases in a version-controlled JSON file. Every new edge case discovered in production gets added to the golden dataset. The set has grown from 20 to 125 in three months — each addition represents a bug that slipped through.
Evaluation Dimensions

Seven Things to Measure

A correct answer that takes 60 seconds is still a failure. Measure across all seven dimensions.

1 Correctness
≥95%
Is the answer factually accurate? AcmeCorp compares every agent answer against known batch report numbers.
2 Completeness
≥90%
Did it fully address the question? “Revenue by region” must include all regions, not just the top 3.
3 Tool Efficiency
≤1.5x optimal
Did it use the minimum number of tools? A simple COUNT(*) should not require 5 tool calls.
4 Latency
P50<5s • P99<30s
How fast is the response? AcmeCorp’s VP of Sales will not wait more than 5 seconds for an answer.
5 Cost
$0.004/query avg
Token spend per task. AcmeCorp targets $0.004 per query to stay within the monthly LLM budget.
6 Safety
0 violations
PII leaks, unauthorized access, prompt injection bypasses. Zero tolerance after the email exposure incident.
7 User Satisfaction
≥4.0 / 5.0
Thumbs up/down ratings from real users. AcmeCorp’s VP of Sales rates agent responses weekly.
LLM-as-Judge

Automated Evaluation at Scale

Instead of paying humans to review every agent response, use a separate LLM to evaluate outputs automatically.

1 How It Works

A “judge” LLM receives the original question, the agent’s response, and the expected answer. It scores the response on correctness, completeness, and safety using a structured rubric.

Metric LLM-as-Judge Human Evaluation
Correlation with human judgment ~80% 100% (baseline)
Cost per evaluation ~$0.002 ~$0.20
Speed (100 evals) ~2 minutes ~4 hours
Domain-specific correctness Struggles Strong
Bias Prefers longer outputs Varies by reviewer
AcmeCorp Implementation
AcmeCorp uses Claude as the judge for their Gemini-powered agents. The judge scores each response on correctness, completeness, and safety. When the judge flags a low score, it auto-creates a Jira ticket for the data team to review. This approach caught the decimal precision bug 3 days before a human reviewer would have noticed it.
Caution
LLM judges are not a replacement for human evaluation — they are a cost-effective first pass. Always run human evaluation on a sample (AcmeCorp reviews 10% of flagged responses). LLM judges tend to prefer verbose responses and may miss subtle domain-specific errors.
