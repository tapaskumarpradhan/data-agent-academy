The ReAct Loop

An AcmeCorp regional manager types: "Show me Q3 revenue by region." Watch the agent think, act, observe, and iterate until it delivers the answer.

Core Concept

THINK → ACT → OBSERVE → Repeat

ReAct (Reasoning + Acting) is the fundamental loop behind every capable agent. Instead of trying to answer in one shot, the agent breaks the problem into steps and iterates.

Think
The agent reasons about what it knows, what it still needs, and what to do next. This is the LLM's internal chain-of-thought.
Act
The agent calls a tool — runs a query, fetches a schema, or transforms data. This is where reasoning meets the real world.
Observe
The agent examines the tool's output. Did it get what it needed? Is there an error? The observation feeds back into the next THINK step.
Trace Walkthrough

Watching the Agent Work

Query: "Show me Q3 revenue by region" — follow each iteration as the agent converges on the answer.

1
Think
"I need to find which tables contain revenue data. Let me discover the schema first."
Act
Call schema_discovery() — list all tables and columns in the data warehouse.
Observe
Found tables: transactions, sales_summary, regions, stores. The sales_summary table has columns: revenue, quarter, region_id.
2
Think
"Good. I can join sales_summary to regions on region_id. Let me write the SQL."
Act
Call query_execute(SELECT r.name, SUM(s.revenue) FROM sales_summary s JOIN regions r ON s.region_id = r.id GROUP BY r.name)
Observe
Got results — but this includes all quarters, not just Q3. Need to add a WHERE filter.
3
Think
"I forgot to filter by Q3. Let me add WHERE quarter = 'Q3' and re-run."
Act
Call query_execute(SELECT r.name, SUM(s.revenue) FROM sales_summary s JOIN regions r ON s.region_id = r.id WHERE s.quarter = 'Q3' GROUP BY r.name ORDER BY SUM(s.revenue) DESC)
Observe
Q3 revenue by region: West $4.2M, East $3.8M, Central $2.9M, South $2.1M. Data looks correct and complete.
4
Think
"I have the Q3 data grouped by region. Time to format a clear response for the user."
Act
Return formatted response to the user.
"Here's your Q3 revenue by region: West leads at $4.2M, followed by East ($3.8M), Central ($2.9M), and South ($2.1M). Total Q3 revenue: $13.0M."
Why It Matters

Single-Shot vs. ReAct

Without ReAct (Single-Shot)
1. Receive query: "Q3 revenue by region"
2. Guess the table name and column structure
3. Generate SQL in one attempt
4. Hope it works
Result: SQL error — wrong table name. User sees "An error occurred." No recovery path.
With ReAct (Iterative)
1. THINK: what tables exist? ACT: discover schema
2. THINK: write query. ACT: execute. OBSERVE: missing filter
3. THINK: add Q3 filter. ACT: re-execute. OBSERVE: correct data
4. THINK: format response. ACT: return answer
Result: Accurate Q3 revenue breakdown. Self-corrected mid-flight. User sees clean data.
Key Insight
Production Reality
Prompt-only ReAct agents can't handle production complexity. The THINK/ACT/OBSERVE loop must be externalized into a proper state machine. LangGraph externalizes state management — persisting the agent's reasoning trace, tool outputs, and retry counts outside the LLM's context window. This makes the loop debuggable, resumable, and auditable.
