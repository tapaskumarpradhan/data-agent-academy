LangGraph Patterns

AcmeCorp's architect needs to decide how the agent's internal logic flows. A single chain won't cut it — real agents need branching, looping, and parallel execution. Four patterns to choose from, each suited to different tasks.

Pattern A

Sequential Chain

The simplest pattern: each step feeds into the next. No branching, no loops. Predictable and easy to debug.

AcmeCorp Example
Customer onboarding: create account → send welcome email → assign team → schedule call. Each step depends on the last. If one fails, the chain stops.
Pattern B

Conditional Branching

The agent classifies the user's intent, then routes to the appropriate specialized path. A diamond decision node splits the flow.

AcmeCorp Example
"Find data" goes to the query agent; "Create report" goes to the report generator. The router decides in milliseconds which path to take.
Pattern C

Loop with Feedback

Generate, validate, fix. The agent retries until it gets it right — or hits a maximum retry count. Essential for SQL generation where the first attempt is often wrong.

AcmeCorp Example
Query generation: try SQL → check validity against the schema → fix if needed → execute. Most queries converge within 2–3 retries.
Pattern D

Parallel Execution

Fan-out from one node to multiple independent tasks, then fan-in to merge results. Dramatically reduces latency when tasks don't depend on each other.

AcmeCorp Example
Get schema, row counts, and sample data — all at once. What would take 3 sequential calls now takes just 1 round-trip. Latency drops from 9 seconds to 3.
Comparison

When to Use Which Pattern

Pattern Use Case Latency Complexity Error Recovery
Sequential Simple pipelines, ETL steps High (additive) Low Fail-fast; retry entire chain
Conditional Intent routing, feature flags Low Medium Fallback to default branch
Loop SQL gen, self-correction, validation Variable Medium Bounded retries; graceful degrade
Parallel Independent sub-tasks, data gathering Low (concurrent) High Partial results; skip failed branches
Key Insight
LangGraph vs LangChain — When to Use Which? ›

LangChain is excellent for prototyping — quick linear chains, simple prompt-template-output patterns. It gets you to a demo fast.

LangGraph is built for production. It adds what LangChain lacks: cycles (retry loops), parallelism (fan-out/fan-in), conditional routing, and externalized state management.

The rule of thumb: if your agent needs any of the four patterns on this page, you need LangGraph. If it's a one-shot chain with no branching or retries, LangChain may suffice.

For AcmeCorp's Data Agent, LangGraph is the clear choice — SQL generation requires retry loops, intent routing requires branching, and context gathering benefits from parallelism.
