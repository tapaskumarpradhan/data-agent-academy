Module 06 — Lesson 01

Context Engineering

Tuesday morning, Maya Chen stared at DataPulse's logs and felt her stomach drop. AcmeCorp's data agent was loading all 500 table schemas into every single prompt — burning 80K tokens before it even started reasoning. The problem was never the model. It was what they were feeding it.

DATAPULSE — CONTEXT WINDOW BUDGET
The Paradigm Shift

Why Prompt Engineering Is No Longer Enough

Raj Patel, AcmeCorp's VP of Analytics, pulled Maya aside after the Tuesday standup. "DataPulse answered the Southwest revenue question with 42% accuracy yesterday. Sofia Torres — Store #47 — called me directly. She said the numbers were so wrong she went back to spreadsheets." Maya had spent weeks perfecting the system prompt, hand-tuning every instruction. But prompt engineering focuses on crafting the right instructions. Context engineering goes further — it is the discipline of building dynamic systems that populate the entire context window with exactly the right information at exactly the right time. For production AI agents, the challenge is not just what you ask the model, but what the model can see when it reasons.

Maya realized that no amount of prompt polish would help if DataPulse was drowning in 80K tokens of schema data before it could think about Sofia's question. The model did not need better instructions — it needed a smarter context pipeline. That insight shifted her team's entire approach, and it should shift yours too.

01
Prompt Engineering
Static instructions written by a human. Works for single-turn, simple tasks. Breaks down with complex agents — as Maya discovered when DataPulse's carefully crafted prompts still produced wrong answers because the context was bloated.
02
Context Engineering
Dynamic system that assembles the full context — instructions, data, history, retrieved knowledge — at runtime for each turn. This is what Maya needed to build: a pipeline that feeds DataPulse only what it needs, when it needs it.
03
Production Reality
A data agent must juggle schemas, conversation history, retrieved docs, reasoning space, and output tokens — all within a fixed window. AcmeCorp's 500+ table PostgreSQL warehouse made this especially brutal.
Core Framework

6 Principles of Context Engineering

Over the next two weeks, Maya and her team rebuilt DataPulse's context pipeline from scratch. Each principle below maps to a specific failure they encountered and fixed at AcmeCorp. Lena Park, the product manager, tracked each change against their accuracy benchmarks.

P1
JIT Context Loading
Load context just-in-time rather than upfront. Instead of stuffing all 500 table schemas from AcmeCorp's PostgreSQL warehouse into every prompt, retrieve only the 3–5 schemas relevant to the current query. When Sofia asks about Store #47's revenue, DataPulse now loads fact_sales, dim_stores, and dim_dates — not the entire catalog.
P2
Context Window Budgeting
Allocate your token budget like a financial budget. Derek Kim, AcmeCorp's CFO, understood this analogy immediately when Maya presented it at the project review: every token spent on schema descriptions is a token not available for reasoning. Set hard limits per category and monitor utilization.
P3
Summarization Checkpoints
In long conversations, periodically summarize earlier turns into compact state. Raj noticed that DataPulse's accuracy cratered after turn 5 of multi-step analyses. The fix: replace 20 turns of dialogue with a structured summary that captures key decisions, resolved ambiguities, and current task state.
P4
Context Isolation via Sub-Agents
Delegate specialized tasks to sub-agents with their own focused context windows. Maya split DataPulse into specialized workers: a schema-lookup agent does not need conversation history; a SQL-generation agent does not need the business glossary. Each sub-agent stays lean.
P5
Scratchpads & External State
Offload intermediate reasoning to external storage. Write partial results, plans, and working memory to a scratchpad outside the context window. Bring results back in only when needed. This was critical for AcmeCorp's complex cross-regional comparisons that required multiple intermediate calculations.
P6
Token-Efficient Tool Design
Design tool inputs and outputs to minimize token consumption. Return structured, compact results rather than verbose descriptions. Maya redesigned DataPulse's schema tool to return column-name abbreviations and omit null fields, cutting schema tokens by 40%.

Performance Threshold

The Context Cliff

Research shows that model performance degrades sharply once retrieved context exceeds approximately 2,500 tokens. This is the "context cliff" — the point where adding more retrieved information actually hurts accuracy instead of helping it. Maya discovered this the hard way: after switching to JIT loading, she initially retrieved 15 schema chunks per query, thinking more was better. Priya Sharma's accuracy audit showed the numbers getting worse. When Maya trimmed retrieval to the top 5 chunks (staying under 2,500 tokens), accuracy climbed back up.

Below the Cliff (< 2,500 tokens)
Optimal Zone
Model can attend to all retrieved context effectively. Accuracy remains high (90%+). Each additional relevant chunk genuinely helps. This is where DataPulse operates now — loading 5–8 lean schema definitions that stay within budget.
Above the Cliff (> 2,500 tokens)
Danger Zone
Relevant information gets lost in noise. Model starts hallucinating or ignoring key facts. More context = worse results. This was DataPulse v1 — dumping AcmeCorp's full 500-table catalog and wondering why Sofia Torres got wrong numbers.
The Silent Killer

Context Rot: When Good Context Goes Bad

Three weeks after Maya's context overhaul, something strange started happening. DataPulse's accuracy — stable at 89% — began slipping. Not dramatically, not all at once. Priya Sharma caught it first during her Friday audit: accuracy had drifted to 83%. The following Friday, 79%. By the fourth week, 71%. "Nothing changed in the code," Maya insisted. But Priya had a different theory: "The code didn't change. The world did."

This is context rot — the gradual degradation of context quality over time, even when the context engineering pipeline itself is working perfectly. Unlike the context cliff (too much context at once), context rot is insidious. The context window looks healthy. The token budget is balanced. But the information inside has silently become stale, contradictory, or misleading.

AcmeCorp Reality Check
Priya traced the accuracy drop to three distinct rot sources that had accumulated over four weeks. Each one was invisible in isolation but devastating in combination. Maya realized that context engineering isn't a one-time build — it requires continuous maintenance, like tending a garden that's constantly growing weeds.

The Three Types of Context Rot

R1
Schema Drift Rot
The underlying data schema changes but the agent's cached knowledge doesn't update. At AcmeCorp, the data team renamed customer_email to contact_email during a Q2 migration. DataPulse's RAG embeddings still referenced the old column name. Queries didn't fail — they returned empty results, and the model confidently reported "no customer email data found" for two weeks before anyone noticed.
R2
Glossary Drift Rot
Business definitions evolve but the semantic layer's embedded definitions don't. AcmeCorp's finance team quietly changed the revenue definition to exclude "pending returns" after a quarterly audit. The RAG system still served the old definition. DataPulse was calculating revenue correctly — by an outdated standard. Derek Kim caught the $340K discrepancy during a board prep review.
R3
Embedding Staleness Rot
The vector embeddings used for retrieval gradually lose alignment with the actual data. New tables, new columns, and new query patterns emerge, but the embedding index reflects a snapshot from weeks ago. When AcmeCorp launched a loyalty program with a new loyalty_members table, DataPulse couldn't find it because it wasn't in the embedding index — leading to hallucinated table names.

Detecting Context Rot

Priya built a "context health dashboard" that became Maya's first check every Monday morning. The key insight: you can't prevent context rot, but you can detect it early and remediate before it compounds.

Freshness Signals

Track the age of every context source fed to the agent:

Schema index age Glossary last-updated Embedding generation date ETL pipeline last-run

At AcmeCorp, Priya set a rule: any context source older than 48 hours triggers a warning. Older than 7 days triggers a hard block — the agent admits "my knowledge of this table may be outdated" rather than serving stale data.

Accuracy Drift Detection

Compare agent outputs against known-good answers on a rolling basis:

Weekly golden-query audit Confidence score trending Empty-result rate spike User correction frequency

Priya runs 50 "golden queries" with known correct answers every Friday. When accuracy drops more than 5% week-over-week, she investigates. The empty-result spike was what first tipped her off to the customer_email rename.

The Context Rot Remediation Pipeline

After the four-week accuracy slide, Maya and Priya built an automated remediation pipeline that runs nightly. It became one of AcmeCorp's most critical pieces of infrastructure — Raj calls it "the immune system for DataPulse."

1
Schema Diff Check

Compare current PostgreSQL schema against last-indexed version. Flag any added, renamed, or dropped columns. At AcmeCorp, this catches 3–5 schema changes per week from the data engineering team's regular migrations.

2
Glossary Sync

Pull the latest metric definitions from AcmeCorp's semantic layer and compare against embedded versions. Re-embed any definitions that changed. This caught the revenue definition update before it could cause another discrepancy.

3
Incremental Re-Embedding

Only re-embed changed or new content rather than regenerating the entire vector index. Maya designed this to run in under 90 seconds for AcmeCorp's typical nightly change volume (10–20 modified entries out of 2,000+).

4
Golden Query Regression

Run 50 golden queries post-remediation and compare results against known-good answers. If accuracy drops below 85%, the pipeline blocks deployment and pages Maya. This has fired twice in three months — both times catching issues before users saw them.

AcmeCorp Results
Since implementing the remediation pipeline, DataPulse's accuracy has held steady between 88–92% with no multi-week drift. Priya's Friday audits went from "putting out fires" to "confirming everything's green." The pipeline catches schema changes within 24 hours instead of weeks. Maya's key lesson: "Context engineering is not a build-once activity. It's an ongoing discipline. The context cliff will kill you on day one. Context rot will kill you on day thirty."
Case Study

AcmeCorp: From 42% to 89% — Maya's Context Overhaul

AcmeCorp Scenario
Week one of DataPulse's deployment was a disaster. The agent had access to AcmeCorp's entire PostgreSQL warehouse — 500+ table schemas, a business glossary with 2,000+ terms, and store managers across 200 locations holding multi-turn conversations asking complex analytical questions. With a 128K context window, Maya's team initially tried loading everything upfront. By conversation turn 5, the context window was full. Accuracy had collapsed to 42%. Sofia Torres started a spreadsheet revolt in the Southwest region. Raj Patel's phone would not stop ringing.

Maya spent the following two weeks rebuilding DataPulse's context pipeline, applying every principle above. Lena Park tracked accuracy daily, sharing results in Slack. The transformation was dramatic:

Before: Naive Loading (Week 1)
All 500 schemas loaded upfront (80K tokens consumed before reasoning). By turn 5, conversation history + schemas exceeded the window. Reasoning space collapsed to under 8%. Accuracy dropped to 42%. Sofia Torres reported getting Southwest region totals that were off by millions. Derek Kim flagged the compute costs — every query was burning maximum tokens for minimum accuracy.
After: Context Engineering (Week 3)
JIT schema loading (5–8 relevant tables per turn, pulled from the star schema). Summarization checkpoints every 6 turns. Schema sub-agent with isolated context. RAG capped at 2K tokens. Accuracy stable at 89% through turn 20+. Sofia Torres went from spreadsheet fallback to running her weekly store reviews entirely through DataPulse. Marcus Webb from Security even approved it for Shopify inventory queries.

Maya implemented a three-tier schema loading system, designed around AcmeCorp's actual query patterns from the first month of deployment. Priya Sharma analyzed 2,400 queries to determine which tables appeared most frequently.

Tier 1 — Always Loaded (3 tables): The core star-schema fact and dimension tables (fact_sales, dim_stores, dim_dates) that appear in 78% of queries. These consume ~3K tokens and are always in context. Sofia Torres's store-level questions almost always hit these three.

Tier 2 — JIT Retrieved (2–5 tables): Retrieved based on the current query via semantic search over table descriptions. Examples: dim_products, dim_customers, promotion_calendar. Each adds ~500–800 tokens. When a regional manager asks about product mix, DataPulse pulls dim_products on the fly.

Tier 3 — On-Demand (remaining 490+ tables): Only loaded when the schema sub-agent explicitly identifies them as needed. These are rare-use tables like vendor_contracts, warehouse_transfers, or the loyalty_members table added in Q2. The sub-agent runs in its own context window, evaluates if the table is relevant, and returns only the needed columns.
