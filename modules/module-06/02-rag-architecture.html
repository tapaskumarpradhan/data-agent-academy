RAG Architecture

Maya Chen's context engineering fix pushed DataPulse from 42% to 78% accuracy. But the following Friday, Priya Sharma ran her weekly audit and found a new class of failures: the agent was hallucinating column names for tables added in Q2. The static prompt knew fact_sales perfectly — but it thought the loyalty program table was called customer_rewards when it was actually loyalty_members. DataPulse needed to retrieve fresh knowledge at query time, not rely on stale prompts.

DATAPULSE — RAG PIPELINE
Architecture Overview

The Four Stages of RAG

Every RAG system follows the same fundamental pipeline. Priya's audit report made the case clearly: DataPulse had been operating with a frozen snapshot of AcmeCorp's schema from January, but the data engineering team had added 14 new tables by July. No amount of prompt engineering could keep up with a warehouse that changes quarterly. The agent needed retrieval-augmented generation — a pipeline to fetch live knowledge and inject it into context at query time. Understanding each stage of this pipeline, and where failures occur, is essential for building reliable retrieval in production.

Stage 1
Ingestion
Documents are chunked, embedded into vectors, and stored in a vector database with metadata. At AcmeCorp, this means schema docs, the business glossary, and Shopify integration specs.
Stage 2
Retrieval
User query is embedded and used to find the most semantically similar chunks from the vector store. When Sofia asks about loyalty program stats, retrieval finds the loyalty_members schema.
Stage 3
Augmentation
Retrieved chunks are formatted and injected into DataPulse's context window alongside the user query, staying within the 2K-token RAG budget Maya established.
Stage 4
Generation
The LLM generates a response grounded in both its training knowledge and the retrieved context — producing SQL that references actual column names, not hallucinated ones.
Critical Insight
Each stage is a potential point of failure. When Priya traced the customer_rewards hallucination, she found the problem started at Stage 1: the new loyalty_members table documentation had never been ingested. Poor chunking in Stage 1 means Stage 2 retrieves irrelevant results. Bad augmentation in Stage 3 means the model ignores good results. Debug RAG from left to right — always start at ingestion.
The Foundation Decision

5 Chunking Strategies Compared

Chunking is where RAG succeeds or fails. Maya learned this when she built AcmeCorp's first RAG pipeline using fixed-size 1,024-token chunks. The result: a single chunk contained half of the fact_sales schema mashed together with the beginning of dim_products. DataPulse retrieved that hybrid chunk and generated SQL that mixed columns from both tables. The strategy you choose determines retrieval quality, and retrieval quality determines answer quality. Here are the five main approaches with benchmark accuracy data:

Strategy How It Works Accuracy Best For
Fixed-Size Split every N tokens regardless of content boundaries ~69% Prototyping only; baseline comparison
Recursive Recommended Split by paragraphs, then sentences, then tokens — preserving semantic boundaries ~75% General purpose; best cost/quality ratio
Semantic Use embeddings to detect topic shifts; split at semantic boundaries 91.9% recall / 54% e2e High-recall needs; research documents
Adaptive Dynamically adjust chunk size based on content density and structure ~87% Mixed-format documents; varied content types
Hierarchical Create parent-child chunk relationships; retrieve at multiple granularities ~80% Long documents; when context around a chunk matters
Recommended Starting Point
Recursive chunking at 256–512 tokens with 10–20% overlap. This is what Maya settled on for AcmeCorp's business glossary after testing all five strategies. It gives you the best balance of retrieval quality, implementation simplicity, and cost efficiency. Optimize from here based on your evaluation metrics.
The 80% Rule
80% of RAG failures trace to chunking decisions. Before tuning your embedding model, reranker, or prompt — fix your chunking. Priya Sharma proved this at AcmeCorp: switching from fixed-size to recursive chunking alone lifted DataPulse's schema-retrieval accuracy from 69% to 75%, with zero changes to any other component.
Implementation Details

Chunking Parameters That Matter

When Maya and Priya sat down to configure AcmeCorp's chunking pipeline, they found three parameters dominated the outcome. Getting these right mattered more than the choice of embedding model.

01
Chunk Size
Too small (<100 tokens): loses context, fragments meaning. Too large (>1000 tokens): dilutes relevance, wastes context window. Sweet spot: 256–512 tokens. At AcmeCorp, Priya found that 400-token chunks for schema docs kept each table self-contained while staying under the context cliff threshold.
02
Overlap
Overlap prevents information loss at chunk boundaries. 10–20% overlap means a 512-token chunk shares 50–100 tokens with its neighbors. Critical for preserving cross-boundary concepts like AcmeCorp's glossary entries that reference related terms (e.g., "sell-through rate" referencing "comparable store sales").
03
Metadata
Attach source, section title, document type, and creation date to every chunk. Metadata enables filtering at retrieval time — when DataPulse retrieves schema docs, it filters by doc_type: schema to avoid pulling glossary entries that happen to mention the same table name.

Semantic chunking achieves 91.9% recall — meaning it finds the right information 92% of the time. But end-to-end accuracy drops to 54%. Maya tested semantic chunking on AcmeCorp's documentation and saw the same gap. Why?

Variable chunk sizes: Semantic boundaries create chunks ranging from 50 to 2,000+ tokens. AcmeCorp's glossary entry for "comparable store sales" generated a 1,800-token chunk that consumed nearly the entire RAG budget by itself.

Over-retrieval: High recall means you retrieve many potentially relevant chunks, but the LLM then struggles to identify the most important information within that larger context. DataPulse would retrieve 8 somewhat-relevant chunks when only 2 were actually needed.

Computational cost: Semantic chunking requires an embedding pass during ingestion to detect topic shifts, adding latency and cost that may not be justified given the e2e accuracy. Derek Kim flagged the additional compute cost during the project review.

The takeaway: high retrieval recall does not automatically translate to high answer accuracy. The LLM's ability to use retrieved context matters just as much.

Case Study

AcmeCorp: Chunking the Data Warehouse Documentation

AcmeCorp Scenario
Priya Sharma cataloged everything that needed to go into DataPulse's RAG index. AcmeCorp's data warehouse documentation included three types of content: table descriptions (what each of 500+ tables in the PostgreSQL warehouse contains), column meanings (business definitions for ambiguous column names like rev_net_adj), and business glossary entries (definitions of terms like "comparable store sales" or "sell-through rate" that store managers like Sofia Torres use daily). Each content type required a different chunking approach. Lena Park scheduled a review for Thursday to validate the strategy before ingestion.
Table Descriptions
One chunk per table
Each table description is a self-contained unit (table name, purpose, key columns, relationships). Chunked as a single unit of 200–400 tokens. No overlap needed — table boundaries are natural semantic boundaries. The star-schema core tables (fact_sales, dim_products, dim_stores, dim_dates) each get their own clean chunk.
Column Meanings
Grouped by table
Column definitions grouped by parent table. Each chunk contains all columns for one table (~300–600 tokens). Metadata includes table name and schema for precise retrieval filtering. This is how DataPulse now correctly maps "revenue" to SUM(total_amount) WHERE status='completed' in fact_sales.
Business Glossary
Recursive, 256 tokens
Glossary entries vary in length (50–500 words). Recursive chunking at 256 tokens with 15% overlap. Each chunk tagged with the term name, business domain, and related terms for cross-referencing. Priya personally validated all 2,000+ glossary entries before ingestion — a two-day effort that prevented dozens of downstream errors.

With the chunking pipeline complete and all three content types ingested, Maya re-ran Priya's original audit queries. The loyalty_members table — the one DataPulse had been calling customer_rewards — now resolved correctly every time. The agent's schema-retrieval accuracy climbed from 78% to 86%. But as they would discover in the next lesson, retrieval accuracy and answer accuracy are two different things entirely.
