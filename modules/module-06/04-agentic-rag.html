Module 06 — Lesson 04

Agentic RAG

Basic RAG retrieves passively. Agentic RAG retrieves intelligently — the agent decides when to retrieve, what to search for, and how to evaluate the results before using them.

AGENTIC RAG — ADAPTIVE RETRIEVAL LOOP
The Shift

From Passive to Active Retrieval

In basic RAG, every user query triggers a retrieval step automatically. In agentic RAG, the agent makes three intelligent decisions about retrieval:

W
WHEN to Retrieve
The agent evaluates whether retrieval is needed at all. Simple questions ("What is ROI?") may not need retrieval. Complex analytical queries definitely do. Follow-up questions might reuse previously retrieved context.
Q
WHAT to Search For
Instead of searching with the raw user query, the agent reformulates it into one or more optimized search queries. "Compare Q3 and Q4 sales by region" becomes two separate searches — one for Q3 and one for Q4.
E
HOW to Evaluate Results
After retrieval, the agent assesses whether the results are sufficient and relevant. If not, it reformulates the query and retrieves again. This self-reflective loop prevents the agent from using low-quality context.

Design Patterns

4 Agentic RAG Patterns

01
Query Planning
Decomposition
The agent analyzes the user query and generates a retrieval plan: which sources to query, in what order, and with what search terms. Complex questions are broken into sub-queries that can be answered independently and then combined. This prevents the "monolithic query" problem where one vague search returns poor results.
02
Adaptive Retrieval
Dynamic Decisions
The agent dynamically decides whether to retrieve based on query complexity and existing context. For follow-up questions in a conversation, it may already have the needed context. For novel questions, it triggers full retrieval. This reduces unnecessary latency and token usage.
03
Multi-Source Retrieval
Source Routing
The agent routes different parts of a query to different knowledge bases. Schema questions go to the schema store. Business term questions go to the glossary. Historical query patterns go to the query log. Results from multiple sources are assembled into a unified context.
04
Self-Reflective RAG
Quality Loop
After retrieval, the agent evaluates: Are these results relevant? Do they contain the information needed to answer the query? If not, it reformulates the search and retrieves again. This "retrieve-evaluate-retry" loop runs up to 2–3 iterations before falling back to a graceful degradation response.
Production Constraint

The 8K Token Rule for RAG Context

Even with agentic RAG's sophisticated retrieval, keep the total assembled RAG context under 8K tokens. This constraint comes from the context cliff (Lesson 01) and the need to preserve reasoning space in the context window.

Why 8K?
Models attend best to the first ~2.5K tokens of retrieved context. With multi-source retrieval, you may have 3–4 sources contributing. Budget 2K tokens per source, keeping total RAG context lean and high-signal.
How to Enforce
Set hard token limits per retrieval source. Use reranking to select only the most relevant chunks. Implement a token counter that truncates or summarizes when the RAG budget is exceeded.
When to Exceed
Only exceed 8K when the task explicitly requires it (e.g., comparing two long documents). In those cases, use a sub-agent with its own context window dedicated to the comparison task.
Case Study

AcmeCorp: Agentic RAG in Action

AcmeCorp Scenario
A regional manager asks: "Compare Q3 and Q4 sales by region." Watch how the agentic RAG pipeline handles this complex, multi-part query — decomposing it, routing to multiple sources, and evaluating results before generating the final answer.
Step 1 — Query Planning
Decompose into Sub-Queries
The agent recognizes this is a comparison query requiring two time periods. It generates two sub-queries: (1) "Q3 sales data by region schema and metrics" and (2) "Q4 sales data by region schema and metrics." It also identifies a need to resolve "Q3" and "Q4" against AcmeCorp's fiscal calendar.
Step 2 — Multi-Source Retrieval
Route to Schema Docs + Business Glossary
Sub-query 1 retrieves from the schema documentation: sales_summary table with revenue_net, region_id, and fiscal_quarter columns. Sub-query 2 retrieves from the business glossary: AcmeCorp's fiscal Q3 is Jul–Sep, Q4 is Oct–Dec. Both results are assembled into a unified context of ~3,200 tokens.
Step 3 — Self-Reflection
Evaluate Retrieved Context
The agent checks: Do I have the table structure? Yes. Do I know the fiscal calendar mapping? Yes. Do I know how to join regions? Yes — store_locations.region joins to sales_summary.region_id. Assessment: context is sufficient. No re-retrieval needed.
Step 4 — Generation
Generate Grounded SQL + Response
With high-quality, validated context, the agent generates a SQL query that correctly uses fiscal quarter dates, joins the right tables, groups by region, and compares Q3 vs Q4. The response includes both the data and a natural-language summary of the comparison.

With basic RAG, the raw query "Compare Q3 and Q4 sales by region" would be sent as a single search to the vector store. The results would likely include:

Problem 1: Generic chunks about "sales" that mention multiple quarters, diluting the Q3/Q4 focus.

Problem 2: No fiscal calendar resolution — the agent would likely default to calendar quarters (Jul–Sep for Q3) but might not verify this against AcmeCorp's fiscal year.

Problem 3: Schema and glossary information mixed together in one retrieval, potentially exceeding the context cliff and causing the model to miss critical column names.

Result: The agent would generate SQL with incorrect date ranges, miss the fiscal calendar adjustment, or use the wrong column names — producing a confidently wrong answer.
