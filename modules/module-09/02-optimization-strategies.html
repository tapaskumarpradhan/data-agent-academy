Module 09 — Lesson 02

8 Cost Optimization Strategies

Maya Chen's war room, Day 1 of the sprint. Eight strategies on the whiteboard, fourteen days on the clock, and a CFO who will kill DataPulse if the bill doesn't drop below $3,000.

MAYA'S WAR ROOM — COST WATERFALL
Maya's War Room — Day 1
Maya wrote eight optimization strategies on the whiteboard in the conference room she had claimed for the sprint. Raj Patel walked in, looked at the list, and shook his head. "We can't do all eight in two weeks." Maya circled three items. "We don't have to. Semantic caching alone could cut 30% — Sofia Torres asks almost the same revenue question every morning from Store #47. Prompt compression could save another 20%. We prioritize by impact and layer them in." She drew a line under each strategy with a target cost next to it. The countdown had started.
01
Prompt Caching
90% input cost reduction Low risk

System prompts, tool definitions, and schema descriptions are identical across requests. DataPulse was sending the full 40-table schema definition with every single query — 3,200 times a day. Maya's first move was to cache these static prefixes so AcmeCorp only paid full price once, then got up to 90% discounts on subsequent requests sharing the same prefix.

90% off
How it works: Structure your prompts so that the static portion (system instructions, tool schemas, few-shot examples) comes first. The provider caches this prefix and charges a fraction for subsequent calls. With Anthropic's prompt caching, cached input tokens cost 10% of the standard rate. For AcmeCorp's system prompt of 8K tokens called 3,200 times/day, that translated to massive savings on the first optimization alone. Key requirement: the cached prefix must be byte-identical across requests — even a single character change invalidates the cache. Maya spent Day 1 refactoring DataPulse's prompt templates to ensure the static prefix was truly static.
02
Model Cascading & Routing
40–70% cost reduction Medium risk

Not every query needs your most powerful (and expensive) model. Maya analyzed AcmeCorp's query logs and found that 65% were simple lookups — things like Sofia Torres checking yesterday's revenue or a store manager asking for a SKU price. These were hitting GPT-4 when a smaller, faster model could handle them perfectly. Maya built a classifier to route simple tasks to GPT-3.5 and reserve GPT-4 for complex reasoning like multi-store trend analyses and what-if scenarios.

40–70%
Routing logic example: Use a lightweight classifier (or even keyword rules) to categorize incoming queries. simple_lookup → Haiku (fast, cheap). multi_step_analysis → Sonnet (balanced). complex_reasoning → Opus (maximum capability). Maya's approach for AcmeCorp: try the cheaper model first and escalate to a more capable model only if confidence is below a threshold or the response fails validation. By Day 5, model routing had dropped DataPulse's running cost from $10,800 down to $6,200.
03
Context Compression
80–90% token reduction Medium risk

Conversation histories grow unbounded. Maya discovered that DataPulse was stuffing the full conversation history into every request — some sessions carried 50K+ tokens of context, most of which was irrelevant. When Raj Patel asked a follow-up question about Q3 trends, the prompt included the entire conversation from his earlier inventory question. Maya implemented aggressive compression: summarize older turns, trim tool response payloads, and only include schema columns relevant to the current question.

80–90%
Three compression techniques Maya deployed: (1) History summarization — after 5 turns, replace older messages with a 200-token summary. (2) Response trimming — tool results often return full JSON payloads; extract only the fields the agent needs. DataPulse had been sending entire SQL result sets back through the context. (3) Schema pruning — instead of sending all 500+ table definitions from AcmeCorp's PostgreSQL warehouse, detect which tables are relevant and include only those. By Day 8, prompt compression brought the running total to $4,100.
04
Dynamic Turn Limits
24% cost reduction Low risk

Without limits, agents can spin in reasoning loops indefinitely. Maya found DataPulse sessions that ran for 15+ turns trying to answer questions it simply could not resolve — burning tokens the entire time. She set maximum reasoning steps per task complexity level: simple tasks get 3 turns, complex analysis gets 8, and anything beyond hits a graceful exit with a partial response and a note to the user.

~24%
Implementation: Tag each incoming query with a complexity class. Set max_turns accordingly: simple=3, moderate=5, complex=8. When the limit is reached, the agent must synthesize whatever it has into a best-effort response rather than silently failing. Maya included the turn count in DataPulse's observability metrics so the team could tune limits over time based on real AcmeCorp usage patterns.
05
Semantic Caching
50–80% for repeated queries Medium risk

This was the strategy Maya had circled first on Day 1. She pulled the query logs and proved it to Raj: Sofia Torres at Store #47 asked a variation of "What were yesterday's revenue numbers?" almost every morning. Across 200 stores, dozens of managers asked semantically identical questions daily. Embedding-based caching detects when a new query is similar enough to a previous one and returns the cached response instantly — zero LLM tokens consumed, sub-100ms latency.

50–80%
How it works: Embed each incoming query using a fast embedding model. Compare against a cache of recent query embeddings. If cosine similarity exceeds a threshold (typically 0.95+), return the cached response. Important: set TTL (time-to-live) based on data freshness requirements. AcmeCorp's sales data from today should not be served from yesterday's cache. Maya set TTL to 4 hours for sales data and 24 hours for slower-moving metrics like supplier performance. She also built in automatic cache invalidation on schema changes.
06
Request Batching
20–40% cost reduction Low risk

When an agent makes multiple small LLM calls in sequence (classify, then extract, then format), batch them into fewer calls with structured output. Maya found that DataPulse was making three separate API calls for every query: one to classify intent, one to select relevant tables, and one to generate SQL. One well-crafted prompt could replace all three, reducing overhead and latency.

20–40%
Example: Instead of three calls — (1) classify query intent, (2) select relevant tables, (3) generate SQL — combine into a single prompt that returns structured JSON: {"intent": "...", "tables": [...], "sql": "..."}. Maya also leveraged batch API endpoints for AcmeCorp's non-real-time workloads — the scheduled reports that Raj and Derek reviewed every Monday — at 50% reduced rates.
07
Output Token Limits
10–25% cost reduction Low risk

Unconstrained agents tend to be verbose. DataPulse was generating 2,000-token explanations when Sofia Torres just needed a revenue number and a one-sentence trend summary. Maya set max_tokens appropriate to each task type. Output tokens are typically 3–5x more expensive than input tokens, so even modest reductions compound quickly across AcmeCorp's 3,200 daily queries.

10–25%
Tuning guide: Measure actual output lengths by task type. Set max_tokens at the 95th percentile of observed lengths plus a small buffer. For SQL generation: 300 tokens is usually sufficient. For data summaries like the ones AcmeCorp's store managers request: 500–800. For multi-step analysis narratives that go to Raj's team: 1,200. Review truncation rates weekly and adjust.
08
Off-Peak & Batch Processing
30–50% cost reduction Low risk

Not every agent task is time-sensitive. Maya identified that 22% of DataPulse's daily load was scheduled reports — the Monday morning dashboard Derek reviewed, the weekly supplier scorecards Lena Park used for product decisions, the daily data quality audits Priya Sharma ran. None of these needed real-time processing. She moved them to overnight batch jobs using batch pricing at 50% reduced rates.

30–50%
What AcmeCorp batches: Daily sales summary reports for all 200 stores, weekly trend analyses for Raj's team, data anomaly detection scans that Priya reviews each morning, and pre-computed answers for the twenty most common questions store managers ask. These run as overnight batch jobs, and the results are cached and served instantly when users ask related questions the next day. By Day 11, batch processing brought AcmeCorp's running total to $2,900.

That question — how to keep costs down permanently — is the subject of the next lesson.
Key Insight
The first two optimizations Maya deployed (semantic caching + model routing) delivered 60% of the total savings with minimal quality risk. She deliberately started with the highest-impact, lowest-risk strategies. Do not attempt all eight simultaneously — measure each layer's impact before adding the next. Maya's sprint worked because she could prove to Raj and Derek exactly how much each optimization contributed.
