Module 10 — Lesson 03

Data Quality & Governance for Agents

An agent that returns stale, wrong, or unvalidated data is worse than no agent at all. Build quality checks into every answer.

DATA QUALITY SAFEGUARDS
The Incident

Friday Afternoon: The $0 Sales Call

It was a Friday afternoon when Sofia Torres called Lena Park from Store #47: "DataPulse says my store did $0 in sales yesterday. We definitely didn't do $0 — I was here. We had a great day." Lena escalated to Priya Sharma immediately. Priya checked AcmeCorp's ETL pipeline logs and found the root cause within minutes: the nightly pipeline had failed overnight. The sales_transactions table in AcmeCorp's PostgreSQL warehouse hadn't been updated in 36 hours. DataPulse didn't know this — it queried stale data and returned a technically correct but practically wrong answer. The query ran, found no records for yesterday, and reported $0.

The following week, another incident. Maya Chen's team ran a schema migration over the weekend, renaming customer_email to contact_email in the customer dimension table. On Monday morning, DataPulse's queries involving customer email started failing silently — returning empty result sets instead of errors. Three store managers reported that their customer lookup feature "just stopped working." Nobody got an error message. The data simply vanished.

Priya's Assessment
Priya presented both incidents to Raj Patel and the team. Her conclusion: "We solved consistency with the semantic layer. Lena solved usability with the UX redesign. But we haven't solved trust. DataPulse will confidently present wrong data if we don't teach it to check its own sources before answering." She proposed three safeguards: freshness checks, schema drift detection, and result validation. Raj approved the work that same day.
Safeguard #1

Agent-Driven Freshness Checks

Priya's first safeguard addressed the $0 incident directly. Before answering any time-sensitive question, DataPulse now verifies that the underlying data is sufficiently recent. A query about "today's sales" is misleading if the last data load was 36 hours ago, as Sofia discovered. The agent checks the last_updated_at timestamp of the relevant table and compares it against a freshness threshold — configured per data source in AcmeCorp's warehouse.

Freshness Validation Pattern

Before executing a query, DataPulse runs a metadata check: SELECT MAX(updated_at) FROM sales_transactions. If the result is more than 24 hours old, the agent warns the user. Priya configured the response Sofia would now see: "The latest sales data I have is from Thursday at 11:45 PM. Yesterday's data hasn't loaded yet. Want me to show you Thursday's numbers, or wait for the data team to fix the pipeline?"

Configurable Thresholds

Different tables have different freshness requirements. Priya defined these for AcmeCorp: real-time inventory data requires updates within the hour, daily sales data must be loaded by 6 AM, and monthly financial summaries are acceptable if loaded within the past day. DataPulse stores these thresholds per data source and checks them before every query. Maya's pipeline monitoring also sends alerts to Priya's team when loads fail.

Safeguard #2

Schema Drift Detection

The customer_email to contact_email rename was a textbook case of schema drift — when tables or columns change unexpectedly. Maya's migration was intentional, but nobody updated DataPulse's configuration. An agent that caches the schema it was trained on will break silently when drift happens, returning errors or, worse, empty results that look like real answers. Priya's second safeguard ensures DataPulse detects these changes before they affect users.

Column Rename (The contact_email Incident)

Maya's team renamed customer_email to contact_email during a weekend migration. DataPulse's semantic layer referenced the old name. Queries didn't crash — they returned empty results, which was even more dangerous.

Silent Failure
Table Removal

A deprecated staging table is dropped, but the agent's join logic still references it. Queries that previously worked now return database errors instead of data.

Breaking
Data Type Change

A VARCHAR column is changed to INTEGER. The agent's SQL works, but string comparisons now produce incorrect results or type errors against AcmeCorp's PostgreSQL warehouse.

Silent Failure
New Columns Added

A new discount_type column is added to fact_sales. Not breaking, but DataPulse is unaware it exists and cannot use it for filtering until Priya updates the semantic layer.

Non-breaking
Priya's Detection Strategy
DataPulse now maintains a snapshot of the expected schema (table names, column names, data types) for all 500+ tables in AcmeCorp's warehouse. On a configurable schedule (or before each query), it compares the current database schema against the snapshot. When Priya's system detects that customer_email no longer exists but contact_email appeared, it sends an alert to Maya's data team Slack channel and tells users: "I've detected a change in the database structure that affects customer lookups. The data team has been notified. I'll hold off on this query until the configuration is updated." No more silent failures. No more managers wondering why customer search "just stopped working."
Safeguard #3

Result Validation & Sanity Checks

Even with a correct query on fresh data with a valid schema, the results might indicate a data problem. Priya's third safeguard ensures DataPulse runs sanity checks on every result before presenting it to the user. The $0 sales incident was her motivating example: Sofia's store was open, had customers, and rang up transactions — $0 revenue was obviously wrong, and DataPulse should have flagged it.

Zero & Negative Checks

Revenue of $0 for a store that is open is almost certainly a data loading issue, not a real result. Negative order counts are impossible. Priya configured DataPulse to flag these anomalies: "Store #47 is showing $0 in sales for Thursday. This seems unlikely for a store that was open. The data may not have loaded correctly — want me to check with the data team?" Sofia would have seen this instead of a confident "$0."

Outlier Detection

If a single store suddenly reports 10x its usual revenue, DataPulse notes the anomaly: "Store #12 is showing $1.4M in revenue this week, which is 10x its usual weekly average. This may reflect a data anomaly or a genuine spike. Want me to check the underlying transactions?" Priya built this by comparing each result against the store's 90-day rolling average from fact_sales.

Row Count Validation

If a query returns zero rows when it should return data (e.g., all 200 stores for this month), DataPulse suspects a data issue rather than returning "no results found." It checks if fact_sales has any recent data at all and reports accordingly: "I expected to find data for 200 stores but found 0 records. This suggests a data loading issue rather than a genuine result."

Trust Signal

Confidence Scoring on Query Results

Priya introduced one more layer of transparency: a confidence score on every answer. Not all answers carry the same reliability. A well-sourced query on fresh data with an exact metric match from Priya's semantic layer warrants high confidence. A query built from inferred intent on day-old data with no exact metric match warrants a lower score. DataPulse now shows this to users so they know how much to trust each answer — something Raj had specifically requested after the $3.4M incident.

Governance

Data Lineage Awareness

Derek Kim, as CFO, had one standing requirement: "For any number DataPulse gives me, I need to know where it came from." Priya built data lineage into every response. DataPulse can now answer: "This revenue number comes from the fact_sales table in AcmeCorp's PostgreSQL warehouse, which is fed by the POS system via Maya's nightly ETL pipeline, last run at 11:45 PM." This transparency builds trust and helps debug discrepancies — exactly what was missing during the $3.4M incident.

01
Source Tracking

Each metric knows its source table in AcmeCorp's star schema, the pipeline that populates it, and the schedule on which it refreshes. Priya maintains this in the semantic layer metadata.

02
Transform Chain

DataPulse can explain intermediate transformations: raw POS data from AcmeCorp's 200 stores is cleaned, deduplicated, and aggregated by Maya's dbt models before reaching fact_sales.

03
Citation in Answers

Answers include a collapsible "data source" section showing which tables and metrics were used. Derek uses this when presenting to the board. Marcus Webb uses it for compliance audits.

Lifecycle

Agent Lifecycle Management

DataPulse is not a one-time deployment. Priya mapped out the full lifecycle after realizing that both the $0 incident and the schema drift could have been caught earlier with proper monitoring. The agent follows a lifecycle from initial build through retirement, with continuous monitoring and improvement at every stage.

DataPulse Lifecycle Stage Details
  • Build: Priya defines metrics in the semantic layer, Maya configures data sources, Marcus sets up guardrails and access controls, and Lena designs the conversation UX. The team writes evaluation tests against AcmeCorp's star schema.
  • Deploy: Shadow mode first (DataPulse answers but Priya reviews before delivery), then canary rollout to Sofia and 10 other power-user store managers, then full production across all 200 stores.
  • Monitor: Track query success rates, response latency (targeting sub-3s), user satisfaction (thumbs up/down), data freshness alerts, and schema drift events. Priya reviews the dashboard weekly.
  • Improve: Analyze failed queries to add new metrics, refine Lena's clarification logic, update the semantic layer for new business terms, and retune prompts. The $0 incident led to freshness checks; the schema drift led to drift detection.
  • Retire: When a data source is decommissioned or a new agent replaces DataPulse, gracefully redirect users and archive query logs for Marcus's compliance requirements.
Organization

Organizational Change Management

Technology is the easy part. Getting 200 store managers to actually use DataPulse — and trust its answers — requires deliberate change management. Lena learned this when adoption stalled at 23 users despite perfect accuracy. Raj and Priya built the organizational framework to support long-term adoption.

Ownership Model

Priya owns the semantic layer and reviews failed queries. Lena owns the UX and clarification logic. Maya owns the data pipelines. Marcus owns security and compliance. Without clear ownership, the agent drifts and degrades — as AcmeCorp almost learned the hard way.

User Training

Lena created short video walkthroughs showing store managers what DataPulse can and cannot do. Sofia Torres recorded a 3-minute demo showing her daily workflow. These outperformed the written documentation that nobody read.

Measuring ROI

Raj tracks reduction in ad-hoc SQL requests to Maya's data team, time-to-answer for business questions, user adoption rate (from 23 to 124 managers after Lena's UX fix), and NPS scores. Derek ties the agent's value to measurable outcomes: $1,847/month operating cost vs. hours of analyst time saved.

Feedback Loops

Every DataPulse answer includes thumbs-up/thumbs-down. Negative feedback routes to Priya for triage. She publishes a monthly "DataPulse health" report showing 91% accuracy, top failure modes, and improvement trends. The CEO reviews it quarterly.

Case Study

AcmeCorp: Priya's Safeguards Prevent the Next Incident

Freshness Check Saves Sofia

Two weeks after deploying the safeguards, another ETL failure occurred on a Tuesday night. On Wednesday morning at 9:00 AM, Sofia asked DataPulse: "What are today's sales so far?" The agent checked sales_transactions.last_updated_at and found the most recent record was from Monday at 11:45 PM. Instead of returning $0 again, DataPulse responded: "Today's sales data hasn't loaded yet (last update: Monday 11:45 PM). Want me to show you Monday's full numbers instead?" Sofia said yes, got useful data, and wasn't misled. Meanwhile, DataPulse had already alerted Maya's team about the stale data.

Schema Drift Caught Before Users Notice

A month later, Maya's team renamed total_amount to order_total in the fact_sales table as part of a planned migration. This time, DataPulse's pre-query schema check detected the change immediately: column total_amount no longer exists, but a new column order_total appeared. Instead of returning empty results or crashing, DataPulse sent an alert to Maya's Slack channel and told users: "I've noticed a change in the database structure that affects revenue calculations. The data team has been notified and I'll resume answering revenue questions once the configuration is updated." Maya updated the semantic layer within the hour. Zero users were impacted.

Outcome
Priya's three safeguards prevented over 40 incorrect answers in the first month of deployment. Store managers like Sofia learned to trust DataPulse because it was honest about its limitations instead of confidently presenting wrong data. Maya's data team received actionable alerts instead of angry support tickets. The combination of Priya's semantic layer (Lesson 1), Lena's UX redesign (Lesson 2), and these quality safeguards transformed DataPulse from a tool that embarrassed AcmeCorp in front of the CEO to one running at $1,847/month, 91% accuracy, and sub-3-second response times across all 200 stores.
