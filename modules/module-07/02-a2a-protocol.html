Module 07 — Lesson 02

Agent-to-Agent Protocol (A2A)

When agents need to collaborate with other agents — not just tools — A2A provides the horizontal communication layer.

A2A — AGENT COLLABORATION NETWORK
Two Complementary Protocols

Vertical vs. Horizontal Communication

DataPulse was no longer alone. By midsummer, Maya Chen had built a second agent — InsightBot — for AcmeCorp’s marketing team, specializing in campaign attribution and customer segmentation. Marcus Webb, Security & Compliance Officer, had also requested a third agent: ComplianceGuard, focused on PII detection, access-log auditing, and regulatory checks across AcmeCorp’s 200 stores.

The trouble surfaced on a Tuesday in August. Sofia Torres asked DataPulse: “Show me which products in Store #47 are underperforming relative to their marketing spend.” DataPulse had the sales data. But marketing attribution — ad spend per SKU, campaign ROI, channel breakdowns — lived exclusively in InsightBot’s domain. DataPulse could not answer the question.

Raj Patel suggested the obvious fix: “Just have DataPulse call InsightBot’s API.” Maya groaned. “That’s the N×M problem all over again, but for agents talking to agents. Three agents today, maybe eight next year. Do we really want every agent hardcoded to every other agent?” The answer was no. MCP solved the vertical problem — agents reaching down to tools. Now they needed a standard for the horizontal problem — agents talking to peer agents. That standard is A2A.

↓
MCP
Vertical — Agent to Tool
DataPulse uses MCP to query PostgreSQL, pull Shopify inventory, read Google Sheets. The agent is always the initiator; tools are passive responders.
↔
A2A
Horizontal — Agent to Agent
DataPulse uses A2A to ask InsightBot for marketing attribution data, or to request ComplianceGuard to verify a query before execution. Both sides are intelligent actors.
Key Distinction
MCP servers are deterministic tool wrappers — AcmeCorp’s PostgreSQL server always returns the same rows for the same query. A2A peers are autonomous agents that can reason, plan, and make decisions — InsightBot might refuse a data request if it violates marketing data policies. The protocols are complementary; a production system like AcmeCorp’s uses both.
Core Concepts

Three Pillars of A2A

01
Agent Cards
Discovery Mechanism

Every A2A-compatible agent publishes a JSON descriptor at /.well-known/agent.json. This card declares the agent’s name, capabilities, supported skills, authentication requirements, and endpoint URL. Maya published Agent Cards for all three of AcmeCorp’s agents — DataPulse, InsightBot, and ComplianceGuard — so each could discover the others automatically.

Like a business card for AI agents

02
Task Lifecycle
State Machine

Every A2A interaction is a Task with a well-defined lifecycle. Tasks move through states: submitted, working, input-required, completed, or failed. Both agents can observe and influence the state. When DataPulse asks InsightBot for marketing data, the request becomes a tracked task with a unique ID that Raj can audit later.

Async-first: supports long-running agent work

03
Capability Routing
Smart Delegation

Agent Cards declare specific skills (e.g., “campaign-attribution”, “pii-detection”). A requesting agent can discover and route tasks to the most capable peer based on declared capabilities. DataPulse does not need to know which agent handles compliance — it routes by skill, and ComplianceGuard answers because its card declares that capability.

No hardcoded agent addresses needed

How A2A Tasks Flow

A2A tasks follow a state machine pattern. Let’s trace Sofia’s original question — “products underperforming relative to marketing spend” — as it flows between DataPulse and InsightBot.

Submitted — DataPulse sends a task to InsightBot
DataPulse creates a task with a message: “Provide marketing spend and campaign attribution data for Store #47 SKUs, fiscal Q3.” The task includes a unique ID, DataPulse’s identity, and the list of SKUs as an artifact. InsightBot acknowledges receipt and the task enters its queue.
Working — InsightBot processes the request
InsightBot reasons about the task, uses its own MCP tools to query the Salesforce marketing database, and begins assembling attribution data per SKU. DataPulse subscribes to streaming updates via Server-Sent Events (SSE) so it can show Sofia a progress indicator rather than a blank screen.
Input-Required — InsightBot asks for clarification
InsightBot pauses: “Do you want paid media only, or should I include organic and email attribution?” DataPulse relays the question to Sofia, who replies “all channels.” DataPulse sends the clarification back, and InsightBot resumes processing. This multi-turn negotiation between agents happens transparently.
Completed / Failed — Terminal states
InsightBot returns a structured artifact with per-SKU marketing spend, channel breakdown, and campaign ROI. DataPulse merges this with its own sales data from PostgreSQL and delivers Sofia her answer: a ranked list of products where marketing spend is high but revenue is low. Failed tasks include error details — for instance, if InsightBot could not access the Salesforce data, it would return a failure with context for debugging.

Deep Dive

Inside an Agent Card

An Agent Card is a JSON document that declares everything another agent needs to start collaborating. Here is what DataPulse’s Agent Card looks like in practice — and what each field means for AcmeCorp’s multi-agent ecosystem.

Identity & Endpoint
name, description, url — who the agent is and where to reach it. DataPulse’s card declares its name as “DataPulse”, describes it as AcmeCorp’s analytics agent, and provides the internal endpoint URL where other agents can submit tasks.
Skills
An array of skill objects, each with an id, name, and description. DataPulse declares skills like “sales-analytics” and “inventory-reporting”. InsightBot declares “campaign-attribution” and “customer-segmentation”. ComplianceGuard declares “pii-detection” and “access-audit”.
Authentication
Declares supported auth schemes (OAuth 2.0, API key, etc.). Marcus Webb required all three AcmeCorp agents to use OAuth 2.0 with mutual TLS for inter-agent communication — ensuring that no unauthorized agent can submit tasks.
Capabilities
Flags for supported features: streaming (SSE updates), pushNotifications (webhook callbacks), and stateTransitionHistory (audit trail). Marcus insisted on stateTransitionHistory for ComplianceGuard so every agent interaction produces a full audit log.
Case Study

AcmeCorp: Three Agents, One Ecosystem

By September — the end of AcmeCorp’s fiscal Q3 — all three agents were live and collaborating via A2A. The transformation was dramatic. Questions that previously required manual data pulls from multiple teams now resolved automatically through agent collaboration.

Before vs. After A2A
Before: Sofia asks DataPulse a cross-domain question. DataPulse says “I don’t have marketing data.” Sofia emails the marketing analyst. Two days later, she gets a spreadsheet. She manually combines it with DataPulse’s output.

After: Sofia asks DataPulse the same question. DataPulse discovers InsightBot via its Agent Card, submits a task, receives structured data in seconds, merges it with sales figures, and delivers a unified answer. Total time: 12 seconds.
DataPulse
Sales Analytics Inventory Reporting

AcmeCorp’s original agent. Connects to PostgreSQL, Shopify, and S3 via MCP. When it encounters a question outside its domain, it discovers peer agents through their Agent Cards and delegates via A2A. Lena Park tracks its delegation patterns to plan future agent capabilities.

InsightBot
Campaign Attribution Customer Segmentation

The marketing team’s agent. Connects to Salesforce CRM and Google Analytics via MCP. Publishes its Agent Card at /.well-known/agent.json declaring campaign-attribution and customer-segmentation skills. Receives tasks from DataPulse and returns structured marketing data.

ComplianceGuard
PII Detection Access Auditing

Marcus Webb’s compliance agent. Both DataPulse and InsightBot route sensitive queries through ComplianceGuard before returning results. It checks for PII exposure, validates access permissions, and logs every decision. Priya Sharma reviews ComplianceGuard’s weekly audit reports for data governance.

Result
AcmeCorp’s three agents handle cross-domain questions that no single agent could answer alone. Adding a fourth agent next quarter — a forecasting specialist Raj has been requesting — requires publishing one Agent Card and deploying one service. The existing agents will discover it automatically through A2A. No rewiring, no custom integration code.
