Glossary of Terms

Every acronym, pattern, and concept covered across all 10 modules of the Data Agent Academy, organized alphabetically for quick reference.

A – E
A2A (Agent-to-Agent Protocol)
Google's open protocol for communication between AI agents across different platforms and vendors. Defines how agents discover each other's capabilities, negotiate tasks, and exchange results through standardized Agent Cards.
Module 04 — Autonomy & Collaboration
AAIF (Anthropic Agent Interoperability Framework)
A framework for building interoperable AI agents that can communicate and collaborate across organizational boundaries, with emphasis on safety and controllability.
Module 04 — Autonomy & Collaboration
ABAC (Attribute-Based Access Control)
An access control model that evaluates attributes (user role, department, time of day, data sensitivity) to determine permissions. More flexible than RBAC for complex data agent scenarios where access depends on multiple contextual factors.
Module 08 — Agent Security
ACP (Agent Communication Protocol)
A protocol defining how agents exchange structured messages, including task requests, status updates, and results. Ensures interoperability between agents built on different frameworks.
Module 04 — Autonomy & Collaboration
Agent Card
A machine-readable JSON document that describes an agent's capabilities, supported input/output formats, and authentication requirements. Used in the A2A protocol for agent discovery and task routing.
Module 04 — Autonomy & Collaboration
AgentOps
The discipline of operating AI agents in production, encompassing observability, CI/CD pipelines, evaluation gates, deployment strategies, and versioning. The agent equivalent of MLOps and DevOps.
Module 05 — AgentOps
Agentic RAG
An advanced retrieval pattern where the agent autonomously decides when to retrieve, what to retrieve, and how to combine multiple retrieval results. Unlike basic RAG, the agent can reformulate queries, chain retrievals, and validate relevance before generating answers.
Module 07 — RAG & Retrieval
ANP (Agent Network Protocol)
A protocol for organizing agents into discoverable networks where they can advertise services and be found by other agents needing those capabilities.
Module 04 — Autonomy & Collaboration
BM25
A keyword-based ranking function used in information retrieval. Scores documents by term frequency and inverse document frequency. Often combined with vector search in hybrid retrieval approaches for RAG systems.
Module 07 — RAG & Retrieval
Canary Deployment
A deployment strategy that routes a small percentage of traffic (e.g., one region's stores) to the new agent version while the majority stays on the current version. If metrics degrade, the canary is rolled back without affecting most users.
Module 05 — AgentOps
Circuit Breaker
A fault-tolerance pattern that stops an agent from repeatedly calling a failing service. After a threshold of failures, the circuit "opens" and returns a fallback response instead of attempting more calls, preventing cascading failures.
Module 04 — Autonomy & Collaboration
Context Cliff
The sharp drop in LLM performance that occurs when the context window is nearly full. Unlike gradual degradation, the model's accuracy can fall off suddenly as token usage approaches the limit, causing missed instructions and hallucinations.
Module 06 — Context Engineering
Context Engineering
The practice of strategically managing what information goes into an LLM's context window. Includes token budgeting, prompt caching, semantic caching, and prioritizing high-value context (schema, instructions, relevant history) over low-value content.
Module 06 — Context Engineering
DAG (Directed Acyclic Graph)
A graph structure with directed edges and no cycles. Used in agent orchestration (LangGraph) to define execution flows where tasks depend on previous task outputs, and in data pipelines (Airflow) to model ETL dependencies.
Module 01 — Architecture Foundation
DVC (Data Version Control)
A version control system for data and ML models that works alongside Git. Tracks large datasets, model files, and pipeline stages so that agent training data and evaluation sets can be versioned alongside code.
Module 05 — AgentOps
Embedding Drift
A gradual degradation in retrieval quality that occurs when the embedding model used to encode documents differs from the model used at query time, or when the underlying data distribution shifts over time, causing previously relevant results to rank lower.
Module 07 — RAG & Retrieval
ETL (Extract, Transform, Load)
A data pipeline pattern that extracts data from source systems, transforms it (cleaning, joining, aggregating), and loads it into a target data warehouse. The data backbone that feeds the tables agents query against.
Module 03 — Persona Design
Evaluation Gate
An automated checkpoint in the CI/CD pipeline that runs a golden dataset against the agent and blocks deployment if accuracy, safety, or performance metrics fall below defined thresholds. The agent equivalent of a test suite that must pass before merge.
Module 05 — AgentOps
F – L
Golden Dataset
A curated set of test queries with known correct answers, covering happy paths, edge cases, and failure scenarios. Used in evaluation gates to measure agent accuracy before deployment. Typically includes 50+ representative queries spanning all supported metric types.
Module 05 — AgentOps
Guardrails
Programmatic constraints that limit what an agent can do. Types include scope checking (environment boundaries), PII masking (redacting sensitive data), rate limiting (preventing runaway queries), output filtering (blocking harmful content), and authorization (role-based access).
Module 03 — Persona Design
HITL (Human-in-the-Loop)
A design pattern where a human reviews and approves agent actions before they execute, particularly for high-stakes operations like modifying production data, sending reports to executives, or making financial decisions.
Module 03 — Persona Design
LLM (Large Language Model)
The foundational AI model that powers an agent's reasoning, natural language understanding, and text generation. In a data agent, the LLM interprets user questions, selects tools, generates SQL, and formats responses. Common examples include GPT-4, Claude, and Gemini.
Module 01 — Architecture Foundation
LLM-as-Judge
An evaluation technique where a separate LLM scores the quality of an agent's responses against reference answers. Used in evaluation gates when human review is too slow, but less reliable than deterministic checks for factual accuracy.
Module 05 — AgentOps
M – R
MCP (Model Context Protocol)
Anthropic's open protocol that standardizes how LLMs connect to external data sources and tools. Provides a universal interface so agents can access databases, APIs, and file systems through a consistent connector pattern.
Module 04 — Autonomy & Collaboration
MLOps
The practice of deploying and maintaining machine learning models in production. AgentOps extends MLOps concepts (versioning, monitoring, CI/CD) to the unique challenges of autonomous agents, including prompt versioning and tool behavior tracking.
Module 05 — AgentOps
Model Cascading
A cost optimization pattern that routes simple queries to smaller, cheaper models and only escalates to larger, more expensive models for complex questions. Reduces average cost per query while maintaining quality on hard tasks.
Module 09 — Cost & Economics
NPS (Net Promoter Score)
A user satisfaction metric measuring how likely users are to recommend the data agent to colleagues. Tracked as part of agent governance to quantify the business value and user adoption of the deployed agent.
Module 10 — Semantic Layer & Agent UX
OPA (Open Policy Agent)
A general-purpose policy engine that externalizes authorization decisions from application code. In agent systems, OPA evaluates whether a given agent action is permitted based on declarative policies written in Rego.
Module 08 — Agent Security
PII (Personally Identifiable Information)
Data that can identify an individual: names, emails, phone numbers, SSNs, addresses. Data agents must mask or redact PII before including it in responses, especially when the requesting user lacks authorization to view individual records.
Module 03 — Persona Design
Prompt Caching
An optimization where the LLM provider caches the tokenized prefix of a prompt (system instructions, schema definitions) so repeated calls with the same prefix only pay for new tokens. Reduces latency and cost for agents with large static context.
Module 06 — Context Engineering
Prompt Injection
A security attack where malicious input is crafted to override an agent's system prompt or instructions. For data agents, this could mean tricking the agent into revealing unauthorized data, executing harmful queries, or bypassing guardrails.
Module 08 — Agent Security
RAG (Retrieval-Augmented Generation)
A pattern that enhances LLM responses by retrieving relevant context from external data sources before generating an answer. For data agents, RAG retrieves schema documentation, metric definitions, and example queries to improve SQL generation accuracy.
Module 07 — RAG & Retrieval
RBAC (Role-Based Access Control)
An access control model that assigns permissions to roles (store manager, analyst, admin) rather than individual users. Each agent persona inherits the permissions of its assigned role, limiting what data and actions it can access.
Module 03 — Persona Design
ReAct (Reasoning + Acting)
An agent reasoning pattern that interleaves thinking (reasoning about the problem) with acting (calling tools). The agent thinks about what to do, acts, observes the result, then reasons about the next step. The core loop powering most LLM-based agents.
Module 01 — Architecture Foundation
ROC (Receiver Operating Characteristic)
A diagnostic curve plotting true positive rate against false positive rate at various thresholds. Used in agent evaluation to measure how well the agent distinguishes between queries it can answer correctly and queries it should escalate or decline.
Module 05 — AgentOps
RRF (Reciprocal Rank Fusion)
A technique for combining results from multiple retrieval methods (e.g., keyword search and vector search) into a single ranked list. Assigns scores based on rank position in each list, producing better results than either method alone.
Module 07 — RAG & Retrieval
S – Z
SSO (Single Sign-On)
An authentication mechanism that allows users to log in once and access multiple applications without re-entering credentials. Data agents inherit user identity from SSO to enforce role-based access without requiring a separate login.
Module 08 — Agent Security
Semantic Caching
A caching strategy that stores agent responses keyed by the semantic meaning of the query rather than exact text. Questions like "Q3 revenue" and "third quarter sales" hit the same cache entry, reducing redundant LLM calls and database queries.
Module 06 — Context Engineering
Semantic Layer
An abstraction between raw database tables and business questions that provides canonical metric definitions. Maps business terms ("revenue") to precise calculations (SUM(total_amount) WHERE status='completed'), ensuring consistent answers across all users and agents.
Module 10 — Semantic Layer & Agent UX
Shadow Mode
A deployment pattern where the new agent version processes real queries in parallel with the current version, but only the current version's answers are delivered to users. Shadow results are logged for comparison, allowing risk-free evaluation on production traffic.
Module 05 — AgentOps
Token Budgeting
The practice of allocating a fixed token budget to each section of the agent's context window (system prompt, schema, conversation history, RAG context, reasoning space). Prevents any single section from consuming the entire context and triggering a context cliff.
Module 06 — Context Engineering
ToT (Tree of Thoughts)
An advanced reasoning pattern where the agent explores multiple reasoning paths simultaneously, evaluates each branch, and selects the most promising one. More thorough than linear ReAct but more expensive in tokens and latency.
Module 01 — Architecture Foundation
Zero Trust
A security model that assumes no implicit trust for any user, agent, or system. Every request is authenticated and authorized regardless of origin. For data agents, this means validating permissions on every query, not just at session start.
